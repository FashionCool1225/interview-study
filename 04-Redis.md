## 一、数据结构与应用场景

### Q1：Redis 的五种基本数据类型及使用场景？

**答：**

| 类型 | 底层简述 | 典型使用场景 |
|------|---------|-------------|
| **String** | SDS（简单动态字符串） | 缓存、计数器、分布式锁、session 共享 |
| **Hash** | ziplist 或 hashtable | 存储对象（用户信息、商品详情），字段级读写 |
| **List** | quicklist（ziplist + 链表） | 消息队列、最新动态列表、分页数据 |
| **Set** | intset 或 hashtable | 标签系统、共同好友、去重（如已读集合） |
| **ZSet (Sorted Set)** | ziplist 或 skiplist + hashtable | 排行榜、延迟队列、带权重的优先级队列 |

面试补充：Redis 还有三种高级类型——**Bitmap**（位操作，签到统计）、**HyperLogLog**（基数统计，UV 估算）、**GeoSpatial**（地理位置，附近的人）。

---

### Q2：Redis 的底层数据结构有哪些？

**答：** Redis 的核心底层数据结构有以下几种：

1. **SDS（Simple Dynamic String）**：Redis 中 String 类型的底层实现。相比 C 原生字符串，SDS 记录了 `len`（已用长度）和 `free`（剩余可用长度），获取字符串长度是 O(1)；支持二进制安全和自动扩容，杜绝了缓冲区溢出。

2. **ziplist（压缩列表）**：一块连续的内存空间，所有节点紧凑排列，没有指针开销。适用于数据量小、元素短的场景。当元素增多或元素变大时会升级为 hashtable 或 skiplist。注意：Redis 7.0 后 ziplist 已被 listpack 替代，解决了连锁更新问题。

3. **skiplist（跳表）**：在有序链表基础上增加多级索引层，查找/插入/删除平均时间复杂度 O(logN)。相比平衡树，跳表实现简单、范围查找更高效，是 ZSet 的核心结构之一。

4. **intset（整数集合）**：一块连续内存，所有元素为整数且有序。当 Set 中元素全为整数且数量较少时使用，支持编码升级（int16 -> int32 -> int64）。

5. **hashtable（哈希表）**：dict 结构，包含两个哈希表 `ht[0]` 和 `ht[1]`，支持渐进式 rehash。Redis 的 Hash、Set、ZSet（配合 skiplist）底层都会用到 hashtable。

---

### Q3：ZSet 的底层实现是什么？跳表的结构能详细说说吗？

**答：**

ZSet 的底层根据数据量有两种编码方式：

- **数据量小时**：使用 ziplist（压缩列表），元素按 score 排序紧凑存储。
- **数据量大时**：升级为 **skiplist（跳表）+ hashtable（哈希表）** 的组合。

**跳表结构详解：**
- 跳表是一个多层的有序链表。最底层（level 0）包含所有元素，形成完整的有序链表。
- 每一层都是下一层的"快速通道"，通过随机化的层数实现类似二分查找的效果。
- 每个节点包含：`value`（成员值）、`score`（排序分数）、`backward`（后退指针，支持反向遍历）、`levels[]` 数组（每层包含前进指针和跨度 span）。
- 跨度 span 用于计算排名（rank），从表头到目标节点路径上所有 span 之和即为排名。

**为什么同时需要哈希表？**
- 跳表支持按 score 范围查询（如 ZRANGEBYSCORE），时间复杂度 O(logN + M)。
- 哈希表支持按 member 查 score（如 ZSCORE），时间复杂度 O(1)。
- 两者互补，缺一不可。

**为什么选跳表而不是红黑树/平衡树？**
1. 跳表实现简单，代码可读性好；
2. 范围查找（ZRANGE）直接沿链表遍历，非常自然；平衡树需要中序遍历，不够直观；
3. 并发修改友好，跳表只需修改相邻节点指针，平衡树需要旋转，影响范围大。

---

### Q4：String 类型在什么编码下使用 SDS？SDS 相比 C 字符串有什么优势？

**答：**

String 类型的编码有三种：`int`（纯数字）、`embstr`（短字符串，<=44字节，一次内存分配）、`raw`（长字符串，使用 SDS）。

SDS 相比 C 原生字符串的优势：
1. **O(1) 获取长度**：C 字符串需要遍历到 `\0`，SDS 直接读 `len` 字段。
2. **杜绝缓冲区溢出**：SDS 的 API 在修改前会检查空间是否足够，不够则自动扩容。
3. **减少内存重分配**：SDS 有空间预分配（修改后 len<1MB 则 free=len，否则 free=1MB）和惰性空间释放（缩短时不立即回收，仅更新 len）。
4. **二进制安全**：SDS 以 `len` 判断内容，不以 `\0` 为结束标志，可以存储任意二进制数据（如图片）。

### Q5：Redis 的渐进式 rehash 是怎么实现的？

**答：**

当哈希表需要扩容或缩容时，Redis 不会一次性将所有数据从旧表迁移到新表（因为数据量大时会阻塞），而是采用**渐进式 rehash**：

1. **触发条件**：负载因子 = 已用节点数 / 哈希表大小。未开启 BGSAVE 时，负载因子 >= 1 触发扩容；开启 BGSAVE 时（COW 机制下写操作会额外复制内存），负载因子 >= 5 才触发扩容。负载因子 < 0.1 触发缩容。
2. **分配新表**：为 `ht[1]` 分配新哈希表，大小为第一个 >= ht[0].used * 2 的 2^n。
3. **设置 rehash 索引**：将 `dict` 的 `rehashidx` 设为 0，表示开始 rehash。
4. **迁移数据**：每次对字典执行增删改查时，顺带将 `ht[0]` 在 `rehashidx` 索引处的链表迁移到 `ht[1]`，然后 `rehashidx++`。后台定时任务也会批量迁移（每次 100 个桶或 1ms 时间，先到为准）。
5. **完成**：所有数据迁移完后，`rehashidx` 设为 -1，释放旧表，将 `ht[1]` 设为 `ht[0]`，新建空的 `ht[1]`。

**注意**：rehash 期间，新增操作直接写入 `ht[1]`，查找操作先在 `ht[0]` 找再到 `ht[1]` 找，删除和更新操作在两个表上都查找。这保证了 rehash 过程中数据不丢失、服务不阻塞。

### Q6：Hash 类型的底层编码有哪些？什么时候会转换？

**答：**

Hash 有两种编码：

1. **ziplist**：当 Hash 的字段数 < 128 且每个字段值 < 64 字节时（可配置 `hash-max-ziplist-entries` 和 `hash-max-ziplist-value`），使用 ziplist 存储。field 和 value 交替排列，紧凑省内存。
2. **hashtable**：超过阈值后自动升级为 hashtable 编码。

**面试补充**：ziplist 的优点是内存连续、CPU 缓存友好；缺点是修改时需要移动后续元素（O(N)），且存在连锁更新问题（前置节点长度变化导致后续所有节点都需要更新 prevlen 字段）。Redis 7.0 用 listpack 替代了 ziplist，从根本上解决了连锁更新。

---

### Q7：List 的底层实现是什么？

**答：**

Redis 3.2 之前，List 根据数据量使用 ziplist 或 linkedlist（双向链表）。

Redis 3.2 之后，统一使用 **quicklist**，它是 ziplist + 双向链表的混合体：
- quicklist 是一个双向链表，每个节点是一个 ziplist（默认大小 8KB，可通过 `list-max-ziplist-size` 配置）。
- 插入/删除时先找到对应 ziplist 节点操作，如果一个 ziplist 过大则分裂。
- 兼顾了链表的灵活性和 ziplist 的内存效率，减少了碎片和指针开销。

Redis 7.0 中，quicklist 内部的 ziplist 被替换为 **listpack**，解决了连锁更新问题。

## 二、持久化

### Q8：RDB 和 AOF 的区别？该怎么选择？

**答：**

| 对比项 | RDB（快照） | AOF（追加日志） |
|-------|------------|----------------|
| **原理** | 某一时刻的全量内存快照 | 记录每一条写命令 |
| **文件大小** | 小（二进制压缩） | 大（文本命令） |
| **恢复速度** | 快（直接加载） | 慢（逐条重放命令） |
| **数据安全** | 可能丢失最后一次快照后的数据 | 取决于 fsync 策略，最多丢 1 秒数据 |
| **性能影响** | fork 时有短暂阻塞，之后无影响 | 每条写命令有磁盘 IO 开销 |
| **适用场景** | 数据备份、灾难恢复、快速迁移 | 数据安全要求高的场景 |

**生产建议**：两者同时开启。Redis 重启时优先加载 AOF（数据更完整）。如果数据允许少量丢失且追求恢复速度，可以只用 RDB。

---

### Q9：RDB 快照是怎么生成的？fork 子进程的原理是什么？

**答：**

RDB 生成过程：

1. **触发**：`save`（阻塞主进程）、`bgsave`（推荐，异步）或配置自动触发条件（如 `save 900 1` 表示 900 秒内至少 1 次写操作则触发）。
2. **fork 子进程**：Redis 主进程调用操作系统的 `fork()` 系统调用，创建一个子进程。fork 期间主线程会阻塞（阻塞时间与页表大小有关，内存越大页表越大，fork 越慢，通常毫秒级）。
3. **写时复制（COW）**：fork 后，父子进程共享物理内存页。当主进程修改某个内存页时，操作系统会复制该页给子进程，子进程仍然读取原始数据。这样子进程就能在不阻塞主进程的情况下，生成一致性快照。
4. **生成 RDB 文件**：子进程遍历内存数据，写入临时 RDB 文件，完成后原子替换旧文件。

**注意事项：**
- fork 时的阻塞：Redis 是单线程模型，fork 期间无法处理请求。大内存实例（如 10GB+）建议关注 fork 耗时。
- COW 的内存开销：如果 bgsave 期间写操作密集，可能导致内存使用量接近翻倍。可通过 `client-output-buffer-limit` 和监控 `latest_fork_usec` 来观察。
- Linux 的 THP（Transparent Huge Pages）会增大 fork 的 COW 开销，建议关闭。

---

### Q10：AOF 持久化的工作原理？AOF 重写是怎么回事？

**答：**

**AOF 工作流程：**
1. **命令追加**：每条写命令执行后，追加到 AOF 缓冲区（内存）。
2. **写入文件**：根据 `appendfsync` 策略刷盘：
   - `always`：每条命令都 fsync，最安全但最慢。
   - `everysec`（默认）：每秒 fsync 一次，兼顾性能和安全，最多丢 1 秒数据。
   - `no`：由操作系统决定何时 fsync，性能最好但可能丢数据。
3. **AOF 重写**：随着时间推移，AOF 文件会越来越大（包含大量冗余命令，如多次 SET 同一个 key）。重写机制会生成一个新的 AOF 文件，只包含将当前数据集恢复所需的最小命令集。

**AOF 重写流程（BGREWRITEAOF）：**
1. fork 子进程。
2. 子进程根据当前内存数据生成新 AOF 文件。
3. 重写期间新的写命令同时追加到 AOF 重写缓冲区（保证数据不丢）。
4. 子进程完成后，将重写缓冲区的内容追加到新 AOF 文件末尾。
5. 原子替换旧 AOF 文件。

---

### Q11：什么是混合持久化？有什么优势？

**答：**

Redis 4.0 引入了混合持久化（`aof-use-rdb-preamble yes`，默认开启）：

- **原理**：AOF 重写时，前半部分是 **RDB 格式的全量快照**（由子进程生成），后半部分是 **AOF 格式的增量命令**（重写期间的新写命令）。
- **优势**：
  - 恢复速度快：RDB 部分加载速度远快于 AOF 逐条重放。
  - 数据完整性好：AOF 部分保证了重写期间的写操作不丢失。
  - 文件体积适中：RDB 部分是压缩的二进制格式，比纯 AOF 命令小很多。

- **缺点**：混合持久化的 AOF 文件前半部分是二进制格式，`redis-check-aof` 工具在修复时只能修复 AOF 部分。

**面试加分点**：Redis 加载混合持久化文件时，先按 RDB 格式加载前半部分，再按 AOF 格式重放后半部分，两者无缝衔接。

### Q12：fork 子进程时为什么会有阻塞？如何优化？

**答：**

**阻塞原因：**
- `fork()` 系统调用需要复制父进程的**页表**（Page Table）。页表大小与进程占用的内存成正比。
- 例如 Redis 占用 10GB 内存，页表可能有几十 MB，复制页表需要几十到上百毫秒。
- 由于 Redis 是单线程模型，fork 期间主线程完全阻塞，无法处理任何请求。

**优化策略：**
1. **控制 Redis 实例内存**：单个实例不超过 10-15GB，避免 fork 时间过长。
2. **关闭 THP（Transparent Huge Pages）**：THP 将页大小从 4KB 扩大到 2MB，虽然减少页表条目，但 COW 时以 2MB 为单位复制，实际内存开销更大。执行 `echo never > /sys/kernel/mm/transparent_hugepage/enabled`。
3. **使用高性能硬件**：SSD、高主频 CPU 可加快 fork 和 COW 速度。
4. **减少 fork 频率**：合理配置 `save` 策略，避免频繁触发 bgsave。
5. **监控 fork 耗时**：通过 `INFO stats` 中的 `latest_fork_usec` 观察，一般建议 < 100ms。

## 三、高可用

### Q13：Redis 主从复制的原理？全量同步和增量同步的区别？

**答：**

**全量同步（Full Resynchronization）**：发生在从节点第一次连接主节点，或者断线时间过长导致复制偏移量差距过大无法增量同步时。

流程：
1. 从节点发送 `PSYNC ? -1` 请求全量同步。
2. 主节点执行 `BGSAVE`，fork 子进程生成 RDB 快照。
3. 生成 RDB 期间的新写命令暂存到 **repl_backlog**（复制积压缓冲区，默认 1MB，环形缓冲区）。
4. RDB 文件通过 Socket 发送给从节点。
5. 从节点加载 RDB 文件（加载期间如果有读请求会返回空或配置返回旧数据）。
6. 主节点将 repl_backlog 中的缓冲命令发给从节点执行。
7. 同步完成，进入命令传播阶段。

**增量同步（Partial Resynchronization）**：发生在主从短暂断线重连后。

流程：
1. 从节点发送 `PSYNC <runid> <offset>`（记录主节点的 runid 和自己的复制偏移量）。
2. 主节点检查 runid 是否匹配，offset 差距是否在 repl_backlog 范围内。
3. 如果满足条件，将 repl_backlog 中 offset 之后的命令发给从节点，完成增量同步。

**关键机制：**
- **复制偏移量（offset）**：主从节点各维护一个 offset，每传播 1 字节 offset+1，通过比较 offset 判断数据是否一致。
- **repl_backlog**：默认 1MB 的环形缓冲区，保存最近传播的数据。如果从节点断线太久，offset 差距超过 backlog 大小，则只能全量同步。建议根据网络稳定性和数据量适当调大。

### Q14：Redis 哨兵（Sentinel）机制是什么？故障转移流程是怎样的？

**答：**

**哨兵的作用：**
哨兵是运行在特殊模式下的 Redis 实例，主要职责：
1. **监控**：定期 PING 主从节点，检测是否存活。
2. **故障转移**：主节点不可用时，自动将从节点提升为新主节点。
3. **通知**：故障转移后，通知客户端新的主节点地址，通知从节点切换主节点。

**故障检测：**
- 哨兵每秒向主从节点和其他哨兵发送 PING。
- **主观下线（SDOWN）**：单个哨兵在 `down-after-milliseconds` 内没收到有效回复，认为节点下线。
- **客观下线（ODOWN）**：对于主节点，需要 `quorum`（配置的哨兵数量）个哨兵都认为主节点主观下线，才能判定为客观下线。这避免了单点误判。

**故障转移流程：**
1. **选举 Leader 哨兵**：所有哨兵通过 Raft 算法选举一个 Leader 来执行故障转移。
2. **选择新主节点**：Leader 按以下优先级选择从节点：
   - 从节点优先级（`replica-priority`，越小越优先）。
   - 复制偏移量最大（数据最新）。
   - runid 字典序最小（兜底规则）。
3. **提升为新主**：Leader 向选中的从节点发送 `SLAVEOF NO ONE`，使其成为新主节点。
4. **切换从节点**：Leader 向其他从节点发送 `SLAVEOF <new-master-ip> <new-master-port>`，让它们复制新主。
5. **通知客户端**：通过发布订阅机制通知客户端新的主节点地址。
6. **处理旧主**：旧主节点恢复后，哨兵会让它作为从节点加入集群。

**部署建议**：至少 3 个哨兵节点，quorum 设为 2，避免脑裂。

### Q15：Redis Cluster 集群的原理？数据是怎么分片的？

**答：**

**数据分片机制：**
Redis Cluster 采用 **哈希槽（Hash Slot）** 进行数据分片，共 16384 个槽。

- 每个 key 通过 `CRC16(key) % 16384` 计算所属槽。
- 每个主节点负责一部分槽，所有主节点覆盖全部 16384 个槽，集群才算上线。
- 客户端发送请求时，如果 key 不在当前节点的槽范围内，节点返回 `MOVED` 重定向到正确的节点。
- 支持 **Hash Tag**：`{user}.name` 和 `{user}.age` 只会对 `{}` 内的内容计算哈希，保证落在同一个槽。

**集群通信——Gossip 协议：**
- 节点之间通过 Gossip 协议交换集群状态信息。
- 常用消息类型：`MEET`（加入集群）、`PING`（心跳检测）、`PONG`（回复）、`FAIL`（节点下线）。
- 每个节点定期随机选择若干其他节点发送 PING，通过多轮传播最终所有节点对集群状态达成一致。
- Gossip 的优点是去中心化、容错性好；缺点是消息传播有延迟，收敛速度慢（相比集中式）。

**故障检测与转移：**
- 如果某个主节点超时未响应，标记为 `PFAIL`（疑似下线）。
- 当超过半数主节点都标记该节点为 PFAIL，状态变为 `FAIL`（确认下线）。
- 该下线主节点的从节点发起选举，获得超过半数主节点投票后晋升为新主节点。

**部署建议**：至少 3 主 3 从（6 个节点）。每个主节点负责约 5461 个槽。

### Q16：Redis Cluster 在扩缩容时如何迁移槽？

**答：**

Redis Cluster 支持在线扩缩容，核心是**槽和数据的迁移**：

1. **添加新节点**：使用 `redis-cli --cluster add-node` 加入集群，此时新节点没有分配任何槽。
2. **重新分片（Reshard）**：使用 `redis-cli --cluster reshard` 命令，将部分槽从现有主节点迁移到新节点。迁移过程：
   - 对要迁移的每个槽，源节点设置槽状态为 `MIGRATING`，目标节点设置为 `IMPORTING`。
   - 源节点遍历该槽下的所有 key，使用 `MIGRATE` 命令逐个迁移到目标节点（原子操作）。
   - 迁移期间，客户端访问正在迁移的 key：源节点如果还有该 key 则正常处理，如果没有则返回 `ASK` 重定向到目标节点。客户端收到 `ASK` 后先发送 `ASKING`，再发送实际命令。
   - 所有 key 迁移完成后，更新集群配置，将槽正式分配给目标节点，`MOVED` 重定向生效。

3. **删除节点**：先将该节点的槽迁移到其他节点，再使用 `redis-cli --cluster del-node` 移除。

**关键点**：迁移是逐个 key 进行的，不是批量，大数据量时迁移耗时较长。可以通过 `CLUSTER SETSLOT ... IMPORTING/MIGRATING` 手动控制。

### Q17：主从复制是同步的还是异步的？如何保证数据不丢失？

**答：**

Redis 的主从复制默认是**异步复制**：主节点执行完命令后直接返回客户端成功，同时异步地将命令传播给从节点。

**异步复制的风险**：如果主节点刚执行完命令就宕机，而命令还没传播到从节点，此时故障转移后数据会丢失。

**保证数据安全的方案：**

1. **`min-replicas-to-write` + `min-replicas-max-lag`**：配置最少从节点数量和最大延迟。例如设置为 1 和 10，表示至少有 1 个从节点且延迟不超过 10 秒时才接受写操作。否则主节点拒绝写入。这降低了数据丢失风险，但不保证零丢失。

2. **`WAIT` 命令**（Redis 3.0+）：`WAIT <numreplicas> <timeout>`，主节点执行写命令后调用 WAIT，阻塞直到指定数量的从节点确认收到该命令或超时。这是**半同步复制**的实现，但注意 WAIT 不保证事务性，如果 WAIT 返回失败，命令可能已经部分传播。

3. **Redis 6.2+ 的客户端缓存和 ACL** 进一步增强了安全性，但 Redis 至今仍没有真正的同步复制（strong consistency），如果需要强一致性，建议使用 **Redlock** 或结合其他方案。

## 四、缓存问题

### Q18：缓存穿透、缓存击穿、缓存雪崩有什么区别？怎么解决？

**答：**

| 问题 | 含义 | 原因 | 解决方案 |
|------|------|------|---------|
| **缓存穿透** | 查询**不存在的数据**，请求直接打到数据库 | 恶意攻击或业务异常，查一个一定不存在的 key | 1. **布隆过滤器**（推荐）：在缓存前置一层过滤器，拦截不存在的 key；2. **缓存空值**：查数据库为空也缓存 null，设置较短过期时间（如 2-5 分钟） |
| **缓存击穿** | **某一个热点 key** 过期瞬间，大量并发请求穿透到数据库 | 热点 key 过期 + 高并发 | 1. **互斥锁（SETNX）**：只允许一个线程查数据库并回填缓存，其他线程等待；2. **逻辑过期**：不设 TTL，在 value 中存逻辑过期时间，过期后异步更新；3. **永不过期**：热点数据设置永不过期，后台异步刷新 |
| **缓存雪崩** | **大量 key 同时过期**或 Redis 宕机，请求全部打到数据库 | 过期时间集中、Redis 故障 | 1. **过期时间加随机值**：避免同时过期，`expire = base + random(1, 300)`；2. **多级缓存**：本地缓存（Caffeine）+ Redis 兜底；3. **限流降级**：使用 Hystrix/Sentinel 对数据库请求限流；4. **集群高可用**：避免 Redis 宕机 |

---

### Q19：布隆过滤器的原理？有什么缺点？

**答：**

**原理：**
布隆过滤器是一个二进制位数组（Bitmap）+ 多个哈希函数。

- **写入**：将 key 经过 k 个哈希函数计算出 k 个位置，将这些位置都置为 1。
- **查询**：将 key 经过同样的 k 个哈希函数，检查对应位置是否全为 1。如果全为 1，key **可能存在**；如果有任何一个为 0，key **一定不存在**。

**优点：** 空间效率极高，不存储 key 本身，适合海量数据判重。

**缺点：**
1. **存在误判率**（False Positive）：可能判断"存在"但实际不存在。误判率与位数组大小、哈希函数个数、数据量有关。可以通过调大位数组和选择合适的哈希函数个数来降低。
2. **不支持删除**：因为不同 key 的哈希位可能重叠，直接置 0 会影响其他 key。可以使用**计数布隆过滤器**（每个位改为计数器）来支持删除，但空间开销增大。

**Redis 实现**：RedisBloom 模块提供了生产级的布隆过滤器实现，支持动态扩容的 Scalable Bloom Filter。

### Q20：缓存与数据库双写一致性怎么保证？

**答：**

这是面试高频题，核心矛盾是：先更新数据库还是先更新缓存？

**常见方案对比：**

| 方案 | 问题 |
|------|------|
| 先更新数据库，再更新缓存 | 并发写时，两个写请求的更新顺序可能和数据库不一致，导致缓存脏数据 |
| 先删缓存，再更新数据库 | 读请求可能在缓存删除后、数据库更新前读取旧值并回填缓存 |
| 先更新数据库，再删缓存（Cache Aside Pattern） | 最常用方案，仍有极端并发问题但概率极低 |

**推荐方案：Cache Aside Pattern（旁路缓存）**
- **读**：先读缓存，命中则返回；未命中则读数据库，写入缓存。
- **写**：先更新数据库，再删除缓存。

**极端问题分析**：先更新数据库再删缓存，在以下极端时序下会出问题——
1. 缓存刚好过期。
2. 读请求读数据库拿到旧值 A。
3. 写请求更新数据库为 B，删除缓存。
4. 读请求将旧值 A 写入缓存。

但这个场景需要"读数据库比写数据库慢"（实际上数据库读通常比写快），所以发生概率极低。

**工程上保证一致性的方案：**
1. **延迟双删**（见下题）。
2. **消息队列重试删除**：删除缓存失败时，将删除操作发送到 MQ 重试。
3. **Canal 订阅 Binlog**：通过 Canal 监听 MySQL Binlog，数据库变更后异步删除/更新缓存。这是**最终一致性**方案，解耦了业务代码，推荐使用。

---

### Q21：延迟双删的原理和注意事项？

**答：**

**延迟双删**是为了解决"先删缓存再更新数据库"方案中，读请求回填旧值的问题。

**流程：**
1. 先删除缓存。
2. 更新数据库。
3. **延迟 N 毫秒后，再次删除缓存**。

**为什么需要延迟？**
- 步骤 1 删缓存后，并发的读请求会查数据库（此时数据库还没更新）并将旧值回填缓存。
- 延迟 N 毫秒后再次删除缓存，确保步骤 2 的写操作已经完成，且回填旧值的读请求也已经执行完毕。

**延迟时间 N 怎么确定？**
- N 应该大于"读请求查数据库 + 写缓存"的耗时。通常为几百毫秒到 1 秒。
- 可以通过监控读请求的 P99 延迟来估算。

**缺点：**
1. 延迟时间不好精确估算，太短可能没覆盖到，太长影响响应。
2. 第二次删除可能失败，仍需重试机制兜底。
3. 增加了写请求的延迟（如果用异步线程执行第二次删除，可以避免）。

**实践建议**：延迟双删 + MQ 重试兜底，或直接使用 Canal 订阅 Binlog 方案更优雅。

---

### Q22：大 Key 问题怎么排查和解决？

**答：**

**什么是大 Key？** 通常指：String 类型 value > 1MB，集合类型元素超过 5000 个，或单个集合占用内存超过 10MB。

**大 Key 的危害：**
1. **阻塞主线程**：Redis 单线程模型，对大 Key 的操作（读取、删除、遍历）耗时较长，阻塞其他请求。
2. **内存不均**：集群模式下大 Key 导致某个节点内存远超其他节点。
3. **网络带宽**：返回大 value 占用大量带宽，影响其他请求。
4. **慢查询**：大 Key 是 Redis 慢查询的主要原因之一。
5. **过期删除阻塞**：大 Key 过期时，Redis 执行删除操作耗时较长。

**排查方法：**
1. `redis-cli --bigkeys`：扫描所有 key，按类型报告最大的 key。
2. `MEMORY USAGE <key>`：估算单个 key 的内存占用。
3. `redis-cli --memkeys`：按内存排序报告 key。
4. RDB 分析工具：如 `redis-rdb-tools`，离线分析 RDB 文件，找出大 Key。
5. 生产环境开启 `slowlog` 监控慢查询，从慢查询日志中发现大 Key。

**解决方案：**
1. **拆分**：将大 Hash/List/Set 拆分为多个小 Key。例如一个大 Hash 有 10 万字段，拆成 10 个 Hash，每个 1 万字段，key 名称加后缀 `{original_key}:0` ~ `{original_key}:9`。
2. **压缩**：String 类型的 value 使用 Snappy/Gzip 压缩后存储。
3. **异步删除**：使用 `UNLINK` 代替 `DEL`，Redis 4.0+ 支持，在后台线程异步释放内存，不阻塞主线程。
4. **避免使用 KEYS 命令**：用 `SCAN` 替代，避免全量扫描阻塞。

### Q23：热 Key 问题怎么排查和解决？

**答：**

**什么是热 Key？** 某个 Key 的访问频率远高于其他 Key，例如秒杀场景下的商品库存 Key。

**热 Key 的危害：**
1. 某个 Redis 节点承受大量请求，CPU/网络成为瓶颈。
2. 集群模式下数据倾斜，热点节点负载远超其他节点。

**排查方法：**
1. **业务预判**：根据业务逻辑预估（如秒杀商品、热点新闻）。
2. **Redis 热点 Key 统计**：Redis 6.2+ 支持 `HOTKEYS` 命令（需开启 LFU 淘汰策略）。
3. **客户端埋点**：在应用层统计 key 访问频率。
4. **`redis-cli --hotkeys`**：需要 Redis 开启 LFU 策略，利用 LFU 计数器找出热 Key。

**解决方案：**
1. **本地缓存**：在应用层使用 Caffeine/Guava Cache 缓存热 Key，减少 Redis 访问。设置较短过期时间（如 5-30 秒），兼顾实时性。
2. **读写分离 + 多从节点**：将热 Key 的读请求分散到多个从节点。
3. **热 Key 复制**：将热 Key 复制多份（如 `product:1001:copy1` ~ `copyN`），分散到不同节点，客户端随机读取。
4. **限流降级**：对热 Key 的访问做限流保护，防止击穿 Redis。

### Q24：如何设计一个多级缓存架构？

**答：**

**多级缓存架构**：L1（本地缓存） -> L2（分布式缓存 Redis） -> DB（数据库）

**架构设计：**
1. **L1 - 本地缓存**（Caffeine/Ehcache）：
   - 存放热数据，容量小（几百到几千条），过期时间短（5-30 秒）。
   - 优点：无网络开销，速度极快（纳秒级）。
   - 缺点：多实例之间数据不一致，内存有限。

2. **L2 - Redis 分布式缓存**：
   - 存放全量热数据和温数据，容量大，过期时间灵活。
   - 优点：多实例共享，数据一致。

3. **DB - 数据库**：
   - 兜底数据源。

**数据流转：**
- 读请求：L1 -> L2 -> DB，任一层命中即返回并回填上层。
- 写请求：更新数据库，删除 L2 缓存，通过 MQ/Redis Pub-Sub 通知所有实例清除 L1 缓存。

**L1 缓存一致性保障：**
- 数据更新时，发布消息到 Redis Pub-Sub 或 MQ，所有应用实例收到消息后清除本地缓存。
- 设置较短的本地缓存过期时间作为兜底。

**适用场景**：读多写少、对延迟敏感的场景，如商品详情页、配置信息。

## 五、其他高频问题

### Q25：Redis 为什么这么快？

**答：**

Redis 高性能的原因是多方面的：

1. **纯内存操作**：数据存储在内存中，读写操作都是内存操作，内存访问速度是磁盘的数万倍。

2. **单线程模型避免上下文切换**：Redis 核心处理逻辑使用单线程，避免了多线程的上下文切换开销和锁竞争。注意：Redis 不是完全单线程，6.0 之后的网络 IO 是多线程的（IO 多线程），但命令执行仍然是单线程。

3. **高效的数据结构**：Redis 针对不同的使用场景设计了专用的底层数据结构（SDS、跳表、压缩列表等），在时间和空间上做了大量优化。

4. **IO 多路复用（epoll）**：使用 epoll 实现 IO 多路复用，单线程可以同时处理大量客户端连接，事件驱动模型避免了阻塞等待。

5. **通信协议简单**：RESP 协议（Redis Serialization Protocol）设计简洁，解析速度快。Redis 6.0 引入 RESP3 支持更多数据类型和客户端缓存。

**面试补充**：Redis 6.0 引入了多线程 IO，解决了网络 IO 瓶颈问题。主线程负责命令执行，多个 IO 线程负责网络数据的读取和写回。命令执行仍然是串行的，所以不需要考虑线程安全问题。

---

### Q26：Redis 的过期策略是什么？内存淘汰策略有哪些？

**答：**

**过期策略：** Redis 采用 **惰性删除 + 定期删除** 的组合策略。

- **惰性删除**：访问 key 时才检查是否过期，过期则删除并返回 null。优点是不浪费 CPU，缺点是过期的 key 如果一直没被访问就一直存在内存中。
- **定期删除**：Redis 每 100ms 随机抽取一批设置了过期时间的 key 检查，删除已过期的。如果过期 key 比例 > 25%，则继续抽取（但有时间上限，避免过度占用 CPU）。

**内存淘汰策略（`maxmemory-policy`）：** 当 Redis 内存达到 `maxmemory` 上限时，按照以下策略淘汰 key：

| 策略 | 范围 | 说明 |
|------|------|------|
| `noeviction` | 不淘汰 | 默认策略，内存满时拒绝写入（返回 OOM 错误） |
| `allkeys-lru` | 所有 key | 淘汰最近最少使用的 key（推荐，最常用） |
| `volatile-lru` | 设置了过期时间的 key | 淘汰最近最少使用的 key |
| `allkeys-lfu` | 所有 key | 淘汰最不经常使用的 key（Redis 4.0+，推荐） |
| `volatile-lfu` | 设置了过期时间的 key | 淘汰最不经常使用的 key |
| `allkeys-random` | 所有 key | 随机淘汰 |
| `volatile-random` | 设置了过期时间的 key | 随机淘汰 |
| `volatile-ttl` | 设置了过期时间的 key | 淘汰 TTL 最短的 key |

**LRU vs LFU：**
- LRU（Least Recently Used）：基于最后访问时间，最近没被访问的优先淘汰。Redis 的 LRU 是近似 LRU，随机抽样 N 个 key（`maxmemory-samples`，默认 5），淘汰其中最久没访问的。
- LFU（Least Frequently Used）：基于访问频率，访问次数最少的优先淘汰。使用 8 位计数器（4 位时间衰减 + 4 位频率对数计数），更准确地反映 key 的热度。

**生产建议**：纯缓存场景用 `allkeys-lfu`；如果部分 key 需要持久化（不设过期时间），用 `volatile-lfu`。

### Q27：Redis 分布式锁怎么实现？Redisson 是怎么做的？

**答：**

**基本实现（SETNX）：**
```
SET lock_key unique_value NX PX 30000
```
- `NX`：key 不存在时才设置（互斥性）。
- `PX 30000`：设置过期时间 30 秒（防止死锁）。
- `unique_value`：使用 UUID 等唯一值（防止误删别人的锁）。
- 释放锁时用 Lua 脚本保证"判断 + 删除"的原子性：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**SETNX 方案的问题：**
1. **锁过期但业务未完成**：锁自动释放后，其他线程可以获取锁，导致并发问题。
2. **主从切换丢锁**：主节点获取锁后还没同步到从节点就宕机，从节点晋升后锁丢失。
3. **不可重入**：同一个线程不能重复获取同一把锁。

**Redisson 的解决方案：**

1. **可重入锁**：使用 Hash 结构，field 为线程标识，value 为重入次数。支持同一线程多次获取。

2. **看门狗（Watchdog）自动续期**：如果获取锁时没指定过期时间（或使用默认 30 秒），Redisson 会启动一个后台线程（看门狗），每隔 `lockWatchdogTimeout / 3`（默认 10 秒）检查锁是否仍被持有，如果是则续期。持有锁的线程释放锁后，看门狗停止续期。这解决了"锁过期但业务未完成"的问题。

3. **RedLock 算法**：解决主从切换丢锁问题。
   - 部署 N 个独立的 Redis 实例（建议 5 个）。
   - 客户端依次尝试在每个实例上获取锁，设置较短超时。
   - 如果在大多数（N/2+1）实例上成功获取锁，且总耗时小于锁的有效时间，则认为获取锁成功。
   - 否则，立即释放所有实例上的锁。
   
   **注意**：RedLock 存在争议（Martin Kleppmann 的质疑文章），Redis 作者 Antirez 也做了回应。生产环境中需要根据业务容忍度决定是否使用。对于不能丢锁的场景，建议结合 ZooKeeper 实现。

### Q28：Redis 事务怎么实现？MULTI/EXEC 和 Lua 脚本有什么区别？

**答：**

**MULTI/EXEC 事务：**
```
MULTI          # 开始事务，后续命令进入队列
SET key1 val1  # 入队，返回 QUEUED
SET key2 val2  # 入队
EXEC           # 原子执行队列中所有命令
```

**特点：**
- **不支持回滚**：与关系型数据库不同，如果事务中某条命令执行失败，其他命令仍然会继续执行。这是 Redis 的设计选择——简单高效。
- **原子性有限**：EXEC 保证命令按顺序执行，不会被其他客户端的命令插入。但不保证所有命令都成功。
- **语法错误 vs 运行时错误**：命令入队时的语法错误（如拼写错误）会导致整个事务被拒绝（Redis 2.6.5+）。运行时错误（如对 String 执行 HGET）只会影响该条命令，其他命令正常执行。

**WATCH 乐观锁：**
```
WATCH key1      # 监视 key1
MULTI
SET key1 val1
EXEC            # 如果 key1 在 WATCH 后被修改，EXEC 返回 nil，事务失败
```

**Lua 脚本：**
```
EVAL "redis.call('set', KEYS[1], ARGV[1]) redis.call('set', KEYS[2], ARGV[2])" 2 key1 key2 val1 val2
```

**Lua 脚本 vs MULTI/EXEC：**

| 对比项 | MULTI/EXEC | Lua 脚本 |
|-------|------------|---------|
| 原子性 | 保证顺序执行，不支持回滚 | 整个脚本原子执行，中间不会被其他命令插入 |
| 灵活性 | 只能执行 Redis 命令 | 可以写逻辑判断、循环等 |
| 性能 | 多次网络往返 | 一次网络往返（推荐 SCRIPT LOAD + EVALSHA） |
| 复杂度 | 简单 | 需要编写和调试 Lua 脚本 |

**推荐**：生产环境优先使用 Lua 脚本，更灵活、性能更好、原子性更强。

### Q29：Redis Pipeline 的作用是什么？和事务有什么区别？

**答：**

**Pipeline（流水线）的作用：**

普通模式下，客户端每发送一条命令都要等待服务端响应（RTT 往返时间）。如果需要执行 1000 条命令，就需要 1000 次 RTT。

Pipeline 将多条命令打包一次性发送给服务端，服务端依次执行后将所有结果一次性返回。

**效果：**
- 1000 条命令，RTT=1ms，普通模式需要 1000ms 网络等待。Pipeline 只需要约 1ms 网络等待（一次往返），总耗时接近 1ms + 1000 条命令的执行时间。
- 网络延迟越大，Pipeline 效果越明显。

**Pipeline vs 事务（MULTI/EXEC）：**

| 对比项 | Pipeline | MULTI/EXEC |
|-------|----------|------------|
| 原子性 | 不保证原子性，命令可能被其他客户端命令穿插 | 保证顺序执行，不会被穿插 |
| 目的 | 减少网络往返，提升吞吐量 | 保证命令的串行执行 |
| 命令数量 | 建议一次打包 500 条左右，太多会占用大量内存 | 无限制，但事务不宜太长 |
| 使用场景 | 批量操作（如批量初始化数据） | 需要原子性的场景（如转账） |

**注意事项：**
- Pipeline 不是 Redis 服务端的特性，而是客户端的实现。服务端只是将命令结果缓存在内存中，最后一次性返回。
- Pipeline 期间服务端会为每个客户端分配一个输出缓冲区（output buffer），大量命令会导致缓冲区占用大量内存。建议每批 500 条左右。
- Pipeline 中的命令不能依赖前一条命令的结果（因为所有命令是一次性发送的）。

### Q30：Redis 的单线程模型能理解一下吗？Redis 6.0 的多线程是怎么回事？

**答：**

**Redis 单线程模型：**
- Redis 的核心命令处理是单线程的：网络事件处理 -> 读取客户端请求 -> 解析命令 -> 执行命令 -> 返回结果，全部在一个线程中串行完成。
- 这里的"单线程"指的是**命令执行阶段**是单线程的，不是 Redis 只有一个线程。Redis 还有后台线程处理持久化（bgsave、bgrewriteaof）、异步删除（UNLINK）、AOF fsync 等任务。

**为什么单线程还快？**
- 瓶颈在网络 IO 和内存访问，不在 CPU。
- 单线程避免了锁竞争和上下文切换。
- 内存操作本身就很快（纳秒级），CPU 不是瓶颈。

**Redis 6.0 多线程 IO：**
- 随着网络硬件性能提升，Redis 的性能瓶颈逐渐从内存转移到网络 IO（read/write 系统调用和协议解析）。
- Redis 6.0 引入了 **IO 多线程**：多个 IO 线程并行处理网络数据的读取、协议解析和结果写回。
- **命令执行仍然是单线程**，所以不需要加锁，编程模型不变。
- 流程：主线程接收连接 -> 分配给 IO 线程读取和解析 -> 主线程串行执行命令 -> IO 线程写回结果。

**配置：**
- `io-threads-do-reads yes`（开启读多线程，默认关闭）。
- `io-threads 4`（IO 线程数，建议 CPU 核数的 3/4）。
- 只在 QPS 很高（>10万）且瓶颈在网络 IO 时才有明显收益。

### Q31：Redis 的内存模型是怎样的？内存占用过高怎么排查？

**答：**

Redis 的内存由以下几部分组成：

1. **数据内存**：存储 key-value 数据的内存，是主要部分。
2. **缓冲内存**：
   - 客户端输出缓冲区（client output buffer）：每个连接一个，暂存要返回给客户端的数据。
   - 复制积压缓冲区（repl_backlog）：主从复制用。
   - AOF 缓冲区：AOF 写入前的暂存区。
3. **内存碎片**：分配器分配内存时的碎片。Redis 默认使用 jemalloc 分配器，碎片率（`mem_fragmentation_ratio` = used_memory_rss / used_memory）通常在 1.0-1.5 之间。

**排查内存占用过高：**
1. `INFO memory`：查看整体内存使用情况，关注 `used_memory`、`used_memory_peak`、`mem_fragmentation_ratio`。
2. `MEMORY USAGE <key>`：查看单个 key 的内存占用。
3. `MEMORY DOCTOR`：Redis 给出内存诊断建议。
4. `redis-cli --bigkeys` / `--memkeys`：找出大 key 和内存占用最高的 key。
5. RDB 分析工具（如 `redis-rdb-tools`）：离线分析，生成内存报告。
6. 检查客户端缓冲区：`CLIENT LIST` 查看每个连接的 `obuf` 大小，排查慢消费者导致的缓冲区膨胀。

**优化建议：**
- 使用合理的编码：小对象使用 ziplist/intset 编码，节省内存。
- 控制 key 和 value 的长度：使用短 key 名称，压缩大 value。
- 设置 `maxmemory` 和合适的淘汰策略，防止 OOM。
- 使用 `ACTIVE DEFRAG`（Redis 4.0+）自动整理内存碎片。

### Q32：如何使用 Redis 实现延迟队列？

**答：**

**方案一：ZSet + 轮询**
- 使用 ZSet，member 为任务 ID，score 为执行时间戳。
- 生产者：`ZADD delay_queue <timestamp> <task_id>`。
- 消费者：定期轮询，`ZRANGEBYSCORE delay_queue 0 <now> LIMIT 0 N`，获取到期的任务，然后用 `ZREM` 删除（配合 Lua 脚本保证原子性）。

```lua
local tasks = redis.call('ZRANGEBYSCORE', KEYS[1], 0, ARGV[1], 'LIMIT', 0, ARGV[2])
for _, task in ipairs(tasks) do
    redis.call('ZREM', KEYS[1], task)
end
return tasks
```

**方案二：Redisson 的 RDelayedQueue**
- Redisson 封装了延迟队列，基于 ZSet 实现，提供了阻塞获取的 API。
- `RBlockingQueue` 配合 `RDelayedQueue`，支持精确延迟和分布式消费。

**方案三：Keyspace Notifications（键空间通知）**
- 利用 Redis 的 key 过期事件通知。设置 key 的过期时间为延迟时间，过期后收到通知执行任务。
- **缺点**：过期事件不保证实时性和可靠性（Redis 重启、事件丢失等问题），不适合对时间精度要求高的场景。

**生产推荐**：ZSet + Lua 脚本方案或 Redisson，可靠性高，支持精确延迟。

### Q33：Redis 的持久化对性能有什么影响？如何优化？

**答：**

**RDB 对性能的影响：**
- `fork()` 子进程时主线程阻塞（大内存可达百毫秒级）。
- COW 机制导致内存使用量可能翻倍。
- 子进程写 RDB 文件产生磁盘 IO。

**AOF 对性能的影响：**
- `everysec` 策略：后台线程每秒 fsync，通常不影响主线程。但如果磁盘 IO 繁忙，fsync 耗时超过 2 秒，主线程会阻塞等待（Redis 的安全机制，防止 AOF 缓冲堆积过大）。
- `always` 策略：每条命令都 fsync，性能最差，QPS 可能降到几百。

**优化建议：**
1. 不要在主节点做 RDB/AOF：将持久化操作放到从节点执行，主节点关闭持久化或仅做 AOF everysec。
2. 合理配置 `save` 策略：避免频繁触发 bgsave。
3. AOF 重写优化：设置 `auto-aof-rewrite-percentage` 和 `auto-aof-rewrite-min-size` 控制重写频率。
4. 使用 SSD：减少磁盘 IO 延迟。
5. 关闭 THP：减少 fork 的 COW 开销。
6. 监控 `latest_fork_usec` 和 `aof_delayed_fsync` 指标。
