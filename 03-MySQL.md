## 一、索引

### Q1：B+树的结构是什么？为什么 MySQL 用 B+树做索引？

**答：**

B+树是一种多路平衡搜索树，它的核心特征是：

**结构特点：**
- 所有数据都存储在叶子节点上，非叶子节点只存索引键值和子节点指针，不存数据。
- 叶子节点之间通过双向链表相连，形成有序链表。
- 每个节点可以包含多个元素，树的层级很矮（通常3-4层就能存储上千万条数据）。
- 所有叶子节点都在同一层，查询性能稳定。

**为什么用 B+树而不是其他数据结构：**

1. **相比 B 树：** B 树的非叶子节点也存数据，导致每个节点能存的索引数更少，树更高，磁盘 IO 次数更多。B+树非叶子节点只存索引，同样大小的磁盘页可以容纳更多索引，树更矮胖，IO 次数更少。另外 B+树叶子节点有链表连接，范围查询非常高效，B 树做范围查询需要中序遍历。

2. **相比哈希表：** 哈希查询是 O(1)，但不支持范围查询、排序、最左前缀匹配。MySQL 大量使用范围查询（`BETWEEN`、`>`、`ORDER BY`），哈希不适用。

3. **相比红黑树/AVL树：** 这些是二叉树，当数据量大时树的高度很高，每个节点是一次磁盘 IO，性能差。B+树每个节点可以有几百上千个子节点，3层B+树就能存几千万数据，只需3次磁盘 IO。

**面试加分点：** InnoDB 中每个 B+树节点默认就是一个 16KB 的数据页。假设主键是 bigint（8字节），指针占6字节，那么一个非叶子节点大约可以存 16KB / (8+6) ≈ 1170 个索引项。3层树可以存 1170 × 1170 × 16（假设一个叶子节点存16条记录）≈ 2000多万条数据，只需要3次磁盘 IO。

### Q2：聚簇索引和非聚簇索引有什么区别？

**答：**

**聚簇索引（Clustered Index）：**
- 聚簇索引的叶子节点存储的是完整的数据行。也就是说索引和数据存在一起。
- InnoDB 中一张表只能有一个聚簇索引。通常是主键，如果没有主键，InnoDB 会选一个唯一非空索引，如果也没有就自动生成一个隐藏的行 ID（rowid）作为聚簇索引。
- 数据在磁盘上的物理存储顺序就是聚簇索引的顺序，所以主键插入是顺序写，性能很好。

**非聚簇索引（Secondary Index，也叫二级索引）：**
- 非聚簇索引的叶子节点存储的是索引列的值 + 对应的主键值，而不是完整的数据行。
- 一张表可以有多个非聚簇索引。
- 通过非聚簇索引查询完整数据时，先查到主键值，然后再到聚簇索引中查找完整数据行，这个过程叫**回表**。

**举个例子：**

假设有一张 `user` 表，`id` 是主键，`name` 上有普通索引。执行 `SELECT * FROM user WHERE name = '张三'`：

1. 先走 `name` 的二级索引，找到 `name = '张三'` 对应的主键 `id = 5`。
2. 拿着 `id = 5` 到聚簇索引中查找，获取完整数据行。这就是回表。

**面试加分点：** 回表是有代价的。如果回表数量太多，MySQL 优化器可能会放弃使用二级索引，直接走聚簇索引全表扫描。这就是为什么有时候明明建了索引却不走索引的原因之一。

### Q3：什么是覆盖索引？

**答：**

覆盖索引是指一个查询语句需要的所有列，都能从二级索引中直接获取，不需要回表到聚簇索引去查完整数据行。

**举例说明：**

表 `user`，主键 `id`，在 `name` 上建了普通索引。

```sql
-- 需要回表，因为要查 *（所有列），二级索引里只有 name 和 id
SELECT * FROM user WHERE name = '张三';

-- 覆盖索引，只需要 id 和 name，这两个都在二级索引的叶子节点里
SELECT id, name FROM user WHERE name = '张三';
```

第二条 SQL 只需要查 `id` 和 `name`，而 `name` 索引的叶子节点已经存了 `name`（索引列值）和 `id`（主键值），所以不需要回表。在 `EXPLAIN` 中会显示 `Using index`，说明使用了覆盖索引。

**覆盖索引的优化场景：**
- 将 `SELECT *` 改为只查需要的字段，就可能利用覆盖索引避免回表。
- 在高并发、大数据量场景下，减少回表能显著提升性能。
- 可以通过建联合索引来实现覆盖索引，把查询条件列和 SELECT 列都包含进索引。

**注意：** 聚簇索引本身就是一种覆盖索引，因为聚簇索引包含了所有列的数据。通常我们说的覆盖索引是针对二级索引的优化。

### Q4：索引失效的场景有哪些？

**答：**

以下是常见的索引失效场景：

1. **对索引列使用函数或运算：**
   ```sql
   -- 索引失效
   SELECT * FROM user WHERE YEAR(create_time) = 2024;
   SELECT * FROM user WHERE id + 1 = 10;
   -- 改写为
   SELECT * FROM user WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
   ```

2. **隐式类型转换：** 字段是 varchar 类型，查询传的是数字，MySQL 会对该列做隐式转换，索引失效。比如 `phone` 是 varchar，写 `WHERE phone = 13800000001` 不走索引。

3. **LIKE 以通配符开头：** `WHERE name LIKE '%张'`，全表扫描。只有 `LIKE '张%'` 才走索引。

4. **OR 条件中部分列没有索引：**
   ```sql
   -- 如果 name 有索引但 age 没有索引，整个条件不走索引
   SELECT * FROM user WHERE name = '张三' OR age = 20;
   ```

5. **联合索引不满足最左前缀原则：** 索引 `(a, b, c)`，查询 `WHERE b = 1 AND c = 2` 不走索引。

6. **NOT IN / NOT EXISTS / IS NOT NULL：** 在某些情况下可能导致索引失效（具体取决于 MySQL 优化器的判断和数据分布）。

7. **索引列参与运算或者使用 != / <>：** MySQL 优化器可能认为全表扫描比走索引更快。

8. **字符集不匹配：** 关联表的字符集不同，MySQL 需要做字符集转换，导致索引失效。

**面试加分点：** 索引是否失效最终还是由 MySQL 优化器根据成本估算决定的。可以用 `EXPLAIN` 来验证实际执行情况。如果优化器判断走索引的成本比全表扫描还高（比如要查的数据占了表的大部分），它会选择全表扫描。

### Q5：联合索引的最左前缀原则是什么？

**答：**

最左前缀原则是指：在联合索引 `(a, b, c)` 中，查询条件必须从索引的最左列开始连续匹配，才能使用索引。

**原理：** 联合索引的 B+树是先按 `a` 排序，`a` 相同时按 `b` 排序，`b` 也相同时按 `c` 排序。所以：
- `a` 列是全局有序的。
- `b` 列只有在 `a` 相同的情况下才有序。
- `c` 列只有在 `a`、`b` 都相同的情况下才有序。

**举例（索引 `(a, b, c)`）：**

| 查询条件 | 是否走索引 | 走索引的列 |
|---|---|---|
| `WHERE a = 1` | 走 | a |
| `WHERE a = 1 AND b = 2` | 走 | a, b |
| `WHERE a = 1 AND b = 2 AND c = 3` | 走 | a, b, c |
| `WHERE b = 2` | 不走 | - |
| `WHERE a = 1 AND c = 3` | 走 | a（c 中间断了） |
| `WHERE b = 2 AND c = 3` | 不走 | - |

**MySQL 5.6+ 的索引下推（Index Condition Pushdown, ICP）优化：**

对于 `WHERE a = 1 AND c = 3` 这种情况，虽然 `c` 不能用于索引匹配，但 MySQL 5.6 之后可以在遍历索引的过程中直接判断 `c = 3` 这个条件，减少回表次数。EXPLAIN 中会显示 `Using index condition`。

**范围查询对最左前缀的影响：**

```sql
-- 索引 (a, b, c)
-- a 是范围查询，b 无法使用索引
WHERE a > 1 AND b = 2 AND c = 3;  -- 只有 a 走索引

-- a 是等值查询，b 是范围查询，c 无法使用索引
WHERE a = 1 AND b > 2 AND c = 3;  -- a, b 走索引
```

遇到范围查询（`>`、`<`、`BETWEEN`、`LIKE`），范围查询列可以用索引，但它右边的列就用不了索引了。

### Q6：如何选择合适的索引？

**答：**

选择索引的核心原则是：**为高频查询的条件列、排序列、分组列建立合适的索引，同时避免过多索引。**

**具体策略：**

1. **选择性高的列优先建索引：** 选择性 = 不同值的数量 / 总行数。性别字段（只有男/女）选择性差，身份证号选择性好。选择性高的列过滤效果好。

2. **查询条件、ORDER BY、GROUP BY 的列考虑建索引：** 这些操作都需要排序或过滤，有索引可以大幅提升性能。

3. **优先使用联合索引而不是多个单列索引：** 一个 `(a, b, c)` 联合索引胜过 `(a)`、`(b)`、`(c)` 三个单列索引。联合索引可以复用（最左前缀），还能实现覆盖索引。

4. **利用覆盖索引避免回表：** 如果查询只需要索引中的列，就不需要回表。在高并发场景下，这一点优化效果非常明显。

5. **控制索引数量：** 一般单表索引不超过 6 个。每个索引在 INSERT/UPDATE/DELETE 时都需要维护，索引太多会影响写入性能，也会占用额外磁盘空间。

6. **字符串考虑前缀索引：** 对于很长的 varchar 字段（如 URL、地址），不需要对整个字段建索引，只对前 N 个字符建前缀索引，节省空间。但要选择合适的长度，保证区分度。

7. **避免冗余索引：** 如果已经有 `(a, b)` 索引，再建 `(a)` 索引就是冗余的。

**面试加分点：** 实际工作中选择索引的流程是：先分析业务中的高频 SQL → 用 EXPLAIN 看执行计划 → 针对慢查询建索引 → 上线后观察慢查询日志验证效果。索引不是一开始就设计完美的，需要根据实际运行情况不断调优。

### Q7：普通索引和唯一索引在性能上有什么区别？

**答：**

**查询性能：** 差别不大。普通索引找到第一个匹配值后继续扫描下一个（因为可能还有重复值），唯一索引找到后直接返回。但实际上通过 B+树定位到具体位置后，两者的查询效率非常接近。

**写入性能：** 有差异。唯一索引在插入时需要额外检查是否存在重复值（需要判断唯一性约束），这个检查过程会增加少量开销。但在 InnoDB 中，由于 MVCC 和 Change Buffer 的存在，差异并不是特别大。

**Change Buffer 的影响：** InnoDB 的 Change Buffer 会缓存二级索引的变更操作（INSERT/UPDATE/DELETE），等后续读操作涉及该页时再合并。普通索引的变更可以进入 Change Buffer，而**唯一索引不能**（因为需要立即判断唯一性），所以唯一索引的写入性能略差于普通索引。

**选型建议：**
- 业务上要求唯一的字段（如手机号、邮箱），用唯一索引，既是约束也是索引。
- 普通业务查询字段，用普通索引。
- 不要因为"觉得唯一索引快"就乱用唯一索引，它的核心价值是数据约束。

### Q8：索引的设计原则有哪些？你在项目中是怎么做索引优化的？

**答：**

**设计原则总结：**

1. **代码先行，索引后上：** 先写好 SQL，用 EXPLAIN 分析执行计划，再决定加什么索引。
2. **联合索引优先于单列索引：** 覆盖更多查询场景，减少索引数量。
3. **选择性高的列放在联合索引左边：** 过滤效果越好，索引利用越充分。
4. **避免过多索引：** 单表一般不超过 6 个索引，写入密集型的表要更少。
5. **避免冗余和重复索引：** 已有 `(a,b)` 就不再需要 `(a)`。
6. **字符串用前缀索引：** 节省空间，但要测试区分度。
7. **利用覆盖索引：** 减少回表，提升查询性能。

**项目中的实践（回答模板）：**

> 在项目中，我主要通过以下方式做索引优化：
> 1. 开启慢查询日志，定位慢 SQL。
> 2. 对慢 SQL 用 EXPLAIN 分析执行计划，关注 type、key、rows、Extra 字段。
> 3. 根据查询模式设计联合索引，尽量让高频查询能走覆盖索引。
> 4. 上线后持续监控慢查询日志，验证优化效果。
> 5. 遇到过一次索引失效的问题，原因是隐式类型转换（字符串字段传了数字），排查后修正了查询参数类型。

## 二、事务

### Q1：事务的 ACID 特性是什么？分别靠什么机制保证？

**答：**

- **A - 原子性（Atomicity）：** 事务中的所有操作要么全部成功，要么全部回滚。靠 **undo log** 实现。事务执行时会将修改前的数据记录到 undo log，如果事务回滚，就用 undo log 恢复数据。

- **C - 一致性（Consistency）：** 事务执行前后，数据库从一个一致状态转变到另一个一致状态（比如转账前后总金额不变）。一致性是最终目标，由原子性、隔离性、持久性来共同保证，同时也需要业务代码的正确性。

- **I - 隔离性（Isolation）：** 多个事务并发执行时，互不干扰。靠 **锁机制 + MVCC** 实现。不同隔离级别提供不同程度的隔离。

- **D - 持久性（Durability）：** 事务一旦提交，数据永久保存，即使宕机也不丢失。靠 **redo log** 实现。MySQL 采用 WAL 策略，先写 redo log 到磁盘，即使宕机也能通过 redo log 恢复数据。

**面试加分点：** 一致性是目的，ACD 是手段。很多时候一致性不仅仅靠数据库，还需要业务层面的保障（比如分布式事务、消息队列最终一致性等）。

---

### Q2：事务隔离级别有哪些？各自解决什么问题？

**答：**

MySQL 有四种隔离级别，从低到高：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| READ UNCOMMITTED（读未提交） | 可能 | 可能 | 可能 |
| READ COMMITTED（读已提交） | 解决 | 可能 | 可能 |
| REPEATABLE READ（可重复读） | 解决 | 解决 | 可能（InnoDB 通过 MVCC+间隙锁基本解决） |
| SERIALIZABLE（串行化） | 解决 | 解决 | 解决 |

**三个问题的含义：**

- **脏读：** 事务 A 读到了事务 B 还未提交的数据。如果 B 回滚了，A 读到的就是脏数据。
- **不可重复读：** 事务 A 两次读同一行数据，中间事务 B 修改并提交了，A 两次读到的值不同。重点是**修改**。
- **幻读：** 事务 A 两次按条件查询，中间事务 B 插入了新行，A 第二次查到了"幻影行"。重点是**新增/删除**。

**InnoDB 的默认隔离级别是 REPEATABLE READ。**

在 RR 级别下，InnoDB 通过 MVCC 解决了不可重复读问题（同一事务内多次读取看到同一快照）。对于幻读问题，在快照读（普通 SELECT）下 MVCC 也能解决，在当前读（SELECT FOR UPDATE 等）下通过**间隙锁（Gap Lock）** 来防止其他事务在范围内插入数据，从而基本解决了幻读。

**面试加分点：** Oracle 默认是 READ COMMITTED，不支持 RR 级别。MySQL InnoDB 默认 RR 级别，实际使用中大部分场景不需要 SERIALIZABLE，因为性能太差。

---

### Q3：MVCC 的实现原理是什么？

**答：**

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 InnoDB 实现高并发读写的核心机制。它让读操作不用加锁，读写不冲突。

**核心组件：**

1. **隐藏列：** InnoDB 为每一行数据添加了两个隐藏列：
   - `DB_TRX_ID`（6字节）：记录最近一次修改该行的事务 ID。
   - `DB_ROLL_PTR`（7字节）：回滚指针，指向 undo log 中该行的上一个版本。

2. **undo log 版本链：** 每次 UPDATE 操作会将旧版本数据写入 undo log，并通过回滚指针将新旧版本串联成链表。链表头部是最新版本，尾部是最早版本。

3. **Read View（读视图）：** 事务在进行快照读时，会生成一个 Read View，包含以下信息：
   - `m_ids`：生成 Read View 时所有活跃（未提交）的事务 ID 列表。
   - `min_trx_id`：活跃事务中最小的事务 ID。
   - `max_trx_id`：系统即将分配给下一个事务的 ID（即当前最大事务 ID + 1）。
   - `creator_trx_id`：创建该 Read View 的事务 ID。

**可见性判断规则：**

对于版本链上的每个版本，用它的 `DB_TRX_ID` 与 Read View 做比较：

- 如果 `trx_id < min_trx_id`：该版本在创建 Read View 之前已提交，**可见**。
- 如果 `trx_id >= max_trx_id`：该版本在创建 Read View 之后才开启，**不可见**。
- 如果 `min_trx_id <= trx_id < max_trx_id`：检查该 `trx_id` 是否在 `m_ids` 列表中。在列表中说明事务还没提交，**不可见**；不在列表中说明已提交，**可见**。

沿着版本链从新到旧找，找到第一个可见的版本就返回。

**RC 和 RR 的区别：**

- **READ COMMITTED：** 每次 SELECT 都会重新生成一个 Read View。所以每次读都能看到最新已提交的数据，导致不可重复读。
- **REPEATABLE READ：** 只在事务第一次 SELECT 时生成 Read View，后续复用同一个 Read View。所以整个事务期间读到的数据是一致的，解决了不可重复读。

### Q4：当前读和快照读的区别是什么？

**答：**

**快照读（Snapshot Read）：**
- 普通的 `SELECT` 语句就是快照读。
- 读取的是 MVCC 版本链中的历史版本（根据 Read View 判断可见性）。
- 不加锁，不会阻塞其他事务的写操作。
- 在 RR 级别下，整个事务期间读到的都是同一个快照（第一次 SELECT 时生成的 Read View）。

**当前读（Current Read）：**
- `SELECT ... FOR UPDATE`（加排他锁）
- `SELECT ... LOCK IN SHARE MODE`（加共享锁）
- `INSERT`、`UPDATE`、`DELETE` 操作
- 读取的是数据的最新版本，并且会对读取到的记录加锁，防止其他事务修改。

**两者对比：**

| | 快照读 | 当前读 |
|---|---|---|
| 读的数据 | 历史版本（MVCC） | 最新版本 |
| 是否加锁 | 不加锁 | 加锁 |
| 是否阻塞写 | 不阻塞 | 阻塞其他事务的写 |
| 典型语句 | `SELECT * FROM t` | `SELECT * FROM t FOR UPDATE` |

**面试加分点：** 在 RR 级别下，快照读能解决幻读（因为始终读同一个快照），但当前读需要靠间隙锁来解决幻读。这就解释了为什么有时候同一个事务里普通 SELECT 和 `SELECT FOR UPDATE` 看到的数据行数不一样。

---

### Q5：什么是间隙锁（Gap Lock）和临键锁（Next-Key Lock）？

**答：**

这两种锁是 InnoDB 在 RR 隔离级别下用来防止幻读的机制。

**记录锁（Record Lock）：**
- 锁定索引上的单条记录。比如 `WHERE id = 5 FOR UPDATE`，锁住 `id = 5` 这一行。

**间隙锁（Gap Lock）：**
- 锁定索引记录之间的"间隙"，防止其他事务在间隙中插入新数据。
- 比如索引上有 `id = 3` 和 `id = 7`，间隙锁会锁住 `(3, 7)` 这个开区间，其他事务不能插入 `id = 4, 5, 6` 的数据。
- 间隙锁不互相冲突，多个事务可以同时对同一个间隙加间隙锁。间隙锁只阻止插入操作。

**临键锁（Next-Key Lock）：**
- 是记录锁 + 间隙锁的组合，锁定一个左开右闭区间 `(a, b]`。
- 比如锁住 `(3, 7]`，既不能修改 `id = 7` 这条记录，也不能在 `(3, 7)` 之间插入新数据。
- **InnoDB 在 RR 级别下默认使用 Next-Key Lock。** 当扫描的索引是唯一索引时，会退化为 Record Lock（因为唯一索引不存在重复值，不需要防幻读）；当扫描的索引是普通索引或范围查询时，使用 Next-Key Lock。

**举个例子：**

```sql
-- 表 user，id 为主键，age 上有普通索引
-- 事务A
SELECT * FROM user WHERE age = 25 FOR UPDATE;
-- 对 age = 25 加 Next-Key Lock

-- 事务B
INSERT INTO user (age) VALUES (25); -- 被阻塞（间隙锁阻止插入）
UPDATE user SET age = 26 WHERE age = 25; -- 被阻塞（记录锁阻止修改）
```

**面试加分点：** 间隙锁是 RR 级别下特有的，在 RC 级别下 InnoDB 会退化为只有记录锁。这也是为什么 RR 能防幻读而 RC 不能的原因。但间隙锁也增加了死锁的概率，因为多个事务可能互相持有对方需要的间隙锁。

### Q6：长事务有什么危害？如何避免？

**答：**

**长事务是指运行时间很长、迟迟不提交的事务。**

**危害：**

1. **undo log 膨胀：** MVCC 依赖 undo log 版本链，长事务不提交，它的 Read View 一直存在，导致旧版本数据无法被 purge 线程清理，undo log 空间持续增长。
2. **连接占用：** 长事务意味着数据库连接长时间被占用，连接池资源被耗尽。
3. **锁等待和阻塞：** 长事务可能持有大量锁，其他事务需要等待，导致整体性能下降。
4. **主从延迟：** 长事务在主库执行很久后才写入 binlog，从库重放时也会出现延迟。

**避免方法：**

1. 确认是否使用了 `autocommit = 1`（默认开启），避免无意中开启长事务。
2. 业务代码中控制事务范围，**尽量小、尽量短**，不要把无关逻辑放进事务。
3. 避免在事务内做耗时操作（网络调用、文件 IO），这些操作应该放到事务外。
4. 设置合理的 `innodb_lock_wait_timeout` 和 `wait_timeout`。
5. 监控长事务：可以通过 `information_schema.innodb_trx` 表查看当前运行中的事务。

```sql
-- 查看当前长事务
SELECT * FROM information_schema.innodb_trx
WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 60;
```

## 三、SQL 优化

### Q1：EXPLAIN 执行计划各个字段是什么含义？

**答：**

使用 `EXPLAIN SELECT ...` 可以查看 SQL 的执行计划，重点关注以下字段：

| 字段 | 含义 | 关注点 |
|---|---|---|
| **id** | 查询的序号 | id 越大优先级越高；id 相同从上往下执行 |
| **select_type** | 查询类型 | SIMPLE（简单查询）、PRIMARY（主查询）、SUBQUERY（子查询）、DERIVED（派生表）、UNION |
| **table** | 访问的表 | - |
| **type** | 访问类型（重要） | 从好到差：system > const > eq_ref > ref > range > index > ALL。**ALL 就是全表扫描，必须优化** |
| **possible_keys** | 可能用到的索引 | - |
| **key** | 实际使用的索引 | NULL 说明没走索引 |
| **key_len** | 使用的索引长度（字节） | 可以判断联合索引用到了几列 |
| **ref** | 与索引比较的列 | const 表示常量等值匹配 |
| **rows** | 预估扫描行数 | 越小越好 |
| **filtered** | 经过条件过滤后的行百分比 | 越高越好，说明过滤效果好 |
| **Extra** | 额外信息（重要） | `Using index`（覆盖索引）、`Using filesort`（需额外排序，要优化）、`Using temporary`（用了临时表，要优化）、`Using index condition`（索引下推） |

**重点关注的几个 type 值详解：**

- **system / const：** 单表最多一行数据，相当于常量查询，最快。const 用于主键或唯一索引的等值查询。
- **eq_ref：** 多表 JOIN 时，被驱动表使用主键或唯一索引等值匹配，每次只返回一行。
- **ref：** 非唯一索引等值查询，可能返回多行。
- **range：** 索引范围扫描（BETWEEN、>、<）。
- **index：** 全索引扫描（扫描整棵索引树），虽然只读索引，但数据量也不小。
- **ALL：** 全表扫描，性能最差。

**key_len 的计算：** 以 varchar(50) 为例，字符集 utf8mb4 每个字符4字节，所以 key_len = 50 × 4 + 2（varchar 长度标识） + 1（NULL 标识） = 203。通过 key_len 可以判断联合索引 `(a, b, c)` 到底用到了哪几列。

### Q2：慢 SQL 优化的思路是什么？

**答：**

慢 SQL 优化有一套标准流程：

**第一步：定位慢 SQL**
- 开启慢查询日志（`slow_query_log = ON`，`long_query_time = 1` 表示超过1秒记录）。
- 使用 `mysqldumpslow` 或 `pt-query-digest` 工具分析慢查询日志，找出最慢、最频繁的 SQL。

**第二步：分析执行计划**
- 对慢 SQL 执行 `EXPLAIN`，重点关注 type 是不是 ALL、key 是不是 NULL、rows 是不是很大、Extra 有没有 Using filesort / Using temporary。

**第三步：针对性优化**

1. **索引优化：**
   - type = ALL：没有走索引，考虑添加索引。
   - key = NULL：索引失效，检查是否有函数操作、隐式转换、LIKE '%xx' 等。
   - 回表太多：考虑覆盖索引，只查需要的字段。

2. **SQL 改写：**
   - 避免子查询，改为 JOIN（子查询会产生临时表）。
   - 避免 `SELECT *`，只查需要的字段。
   - 大 IN 列表考虑改为 JOIN 或分批查询。
   - 用 `UNION ALL` 代替 `UNION`（UNION 需要去重排序）。

3. **表结构优化：**
   - 选择合适的数据类型（能用 int 不用 varchar，能用 tinyint 不用 int）。
   - 大字段（text/blob）考虑垂直拆分到单独的表。

4. **架构层面优化：**
   - 引入缓存（Redis），减少数据库查询。
   - 读写分离，把查询压力分摊到从库。
   - 分库分表（数据量超大时）。

**第四步：验证效果**
- 优化后再次用 EXPLAIN 确认执行计划变好了。
- 持续观察慢查询日志，确认该 SQL 不再出现。

### Q3：大表优化方案有哪些？什么时候需要分库分表？

**答：**

**优化层次（从小到大）：**

1. **SQL 和索引优化：** 最基础也是最先做的，确保每条 SQL 都走了合适的索引。

2. **表结构优化：**
   - 垂直拆分：把不常用的大字段（text、blob）拆到扩展表，减小主表行大小，提高 Buffer Pool 缓存命中率。
   - 水平拆分：按行拆分到结构相同的多个表。

3. **引入缓存：** 热数据放 Redis，减少数据库压力。读多写少的场景效果最好。

4. **读写分离：** 主库负责写，从库负责读。通过主从复制同步数据。

5. **分库分表：** 当单表数据量超过一定阈值（通常经验值是 500万 - 1000万行，或数据文件超过 10GB），且以上手段都无法满足性能要求时，考虑分库分表。

**分库分表策略：**

- **垂直分库：** 按业务功能拆分，比如用户库、订单库、商品库。对应微服务的思想。
- **水平分表：** 同一个库内，把一张大表拆成多张小表（如 `order_0`, `order_1`, ...），通过分片键（如 user_id）决定数据路由到哪张表。
- **水平分库：** 多个库，每个库存放部分数据。

**分片键选择：** 选择查询最频繁的字段作为分片键（比如订单表用 user_id），确保大部分查询不需要跨库。

**分库分表带来的问题：**

1. **分布式事务：** 跨库操作需要分布式事务（Seata、TCC、消息队列最终一致性等方案）。
2. **跨库 JOIN：** 无法直接 JOIN，需要通过应用层组装数据，或者使用宽表/数据冗余。
3. **全局唯一 ID：** 不能用自增 ID，需要用雪花算法（Snowflake）或号段模式生成全局唯一 ID。
4. **聚合查询和排序：** 分页、排序、聚合操作需要在应用层做归并。
5. **数据迁移和扩容：** 增加节点后需要数据重新分布，通常用一致性哈希或者翻倍扩容法。

**常用中间件：** ShardingSphere（Sharding-JDBC / Sharding-Proxy）、MyCat。

### Q4：深分页问题是什么？怎么解决？

**答：**

**问题描述：**

```sql
-- 查询第 100 万页的数据
SELECT * FROM orders ORDER BY id LIMIT 10000000, 10;
```

这条 SQL 看似只取 10 条数据，但 MySQL 实际上需要扫描前 10000010 条数据，丢弃前 10000000 条，只返回最后 10 条。数据量越大、页数越深，性能越差。

**原因：** `LIMIT offset, size` 中的 offset 很大时，MySQL 需要扫描大量数据然后丢弃，非常低效。

**解决方案：**

**方案一：延迟关联（Deferred Join）**

```sql
-- 先通过覆盖索引查出目标行的主键（只扫描索引，很快）
SELECT * FROM orders t1
INNER JOIN (
  SELECT id FROM orders ORDER BY id LIMIT 10000000, 10
) t2 ON t1.id = t2.id;
```

子查询只查 `id` 列，走覆盖索引，避免了大量回表。然后再用这 10 个 id 去主表取完整数据。

**方案二：基于游标的分页（推荐）**

```sql
-- 前端记录上一次查询最后一条记录的 id
SELECT * FROM orders WHERE id > #{lastMaxId} ORDER BY id LIMIT 10;
```

每次查询都带上上一页最后一条记录的 ID，利用主键索引直接定位，性能稳定，不受页数影响。**这种方式要求分页是顺序翻页，不能跳页。** 适合"下拉加载更多"的场景。

**方案三：限制最大页数**

业务上限制最多翻到第 N 页（比如100页），超过后提示用户缩小搜索条件。很多搜索引擎也是这么做的（Google 最多翻到几十页）。

**方案四：使用搜索引擎**

复杂的分页搜索场景可以考虑用 Elasticsearch，ES 的 `search_after` 机制天然适合深分页。

### Q5：JOIN 查询的优化方法有哪些？

**答：**

**1. 小表驱动大表：**

MySQL 的 JOIN 执行通常使用 Nested-Loop Join，外层循环是小表，内层循环是大表。小表做驱动表可以减少内层循环次数。

```sql
-- 如果 A 表数据少，B 表数据多
SELECT * FROM A JOIN B ON A.id = B.a_id;
-- A 是驱动表（小表），效率高
```

**2. 被驱动表的关联字段加索引：**

在 Nested-Loop Join 中，驱动表的每一行都要到被驱动表去匹配，如果被驱动表的关联字段有索引（ref 访问），效率会大幅提升。没有索引的话就变成全表扫描（ALL），非常慢。

**3. 避免不必要的 JOIN：**

如果 JOIN 只是为了获取一两个字段，考虑用冗余字段或应用层组装来代替。

**4. STRAIGHT_JOIN 强制驱动顺序：**

当 MySQL 优化器选择的驱动表不是最优时，可以用 `STRAIGHT_JOIN` 强制指定。但一般不推荐，因为优化器通常能做出正确判断。

**5. 减少 JOIN 的表数量：**

JOIN 的表越多，优化器的排列组合越大，执行计划越可能不是最优。一般建议 JOIN 的表不超过 3 张。

### Q6：COUNT(*) 的优化方式有哪些？

**答：**

**`COUNT(*)` 的语义：** 统计查询结果集中的总行数（包括 NULL 行）。

**不同存储引擎的实现：**

- **MyISAM：** 维护了一个总行数的元数据，`COUNT(*)` 直接 O(1) 返回，非常快。但如果有 WHERE 条件就不行了。
- **InnoDB：** 由于 MVCC 的存在，不同事务看到的行数可能不同，所以 InnoDB 没有维护总行数的元数据。每次 `COUNT(*)` 都需要扫描数据，比较慢。

**优化方式：**

1. **用 `COUNT(*)` 而不是 `COUNT(列名)` 或 `COUNT(1)`：** `COUNT(*)` 是 SQL 标准中统计行数的语法，MySQL 对它做了专门优化。`COUNT(列名)` 需要判断该列是否为 NULL，效率反而可能更差。`COUNT(1)` 和 `COUNT(*)` 在 InnoDB 中性能一样，但 `COUNT(*)` 是标准写法。

2. **自行维护计数：** 对于需要频繁统计的场景，可以用一张计数表或者 Redis 的计数器来维护。写入时更新计数，查询时直接读。要注意并发安全（用原子操作或 CAS）。

3. **利用缓存近似计数：** 如果不需要精确值（如网站的"约 xx 条结果"），可以用 Redis HyperLogLog 做近似计数。

4. **EXPLAIN 估算：** `EXPLAIN` 的 rows 字段可以粗略估计行数，但不精确。

## 四、锁机制

### Q1：InnoDB 的行锁、表锁、间隙锁有什么区别？

**答：**

**表锁（Table Lock）：**
- 锁住整张表，开销小，加锁快，不会出现死锁。
- 并发度低，适合读多写少的场景。
- InnoDB 在 DDL 操作（ALTER TABLE）、`LOCK TABLES` 语句时使用表锁。
- 意向锁（Intention Lock）是一种特殊的表锁，用来协调行锁和表锁的关系。比如加行锁前先加意向排他锁（IX），这样其他事务要加表锁时只需要检查意向锁，不用逐行检查。

**行锁（Row Lock）：**
- 锁住具体的行记录，并发度高，但开销大，加锁慢，可能出现死锁。
- InnoDB 的行锁是加在**索引**上的，不是加在数据行上。如果 SQL 没有走索引，InnoDB 无法通过索引定位行，会退化为**全表扫描 + 对每一行加行锁**，效果等同于表锁，非常影响性能。
- 行锁类型：
  - **记录锁（Record Lock）：** 锁住单行索引记录。
  - **间隙锁（Gap Lock）：** 锁住索引记录之间的间隙，防止插入。
  - **临键锁（Next-Key Lock）：** 记录锁 + 间隙锁，锁住一个范围。RR 级别下默认使用。

**关键结论：** 行锁变表锁的坑一定要避免——UPDATE/DELETE 语句的 WHERE 条件一定要走索引，否则会锁全表。

---

### Q2：乐观锁和悲观锁是什么？在 MySQL 中怎么实现？

**答：**

**悲观锁：**
- 假设数据一定会发生并发冲突，所以在操作数据前就加锁，其他事务必须等待。
- MySQL 中通过 `SELECT ... FOR UPDATE`（排他锁）或 `SELECT ... LOCK IN SHARE MODE`（共享锁）实现。
- 适用场景：写冲突频繁、数据一致性要求高的场景（如金融、库存扣减）。
- 缺点：并发度低，可能造成锁等待和死锁。

```sql
-- 悲观锁示例：扣减库存
BEGIN;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 加排他锁
-- 在应用层判断 stock 是否够
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

**乐观锁：**
- 假设数据一般不会发生并发冲突，只在更新时检查数据是否被其他事务修改过。
- MySQL 中通常通过**版本号字段**实现。

```sql
-- 表里增加 version 字段
ALTER TABLE product ADD COLUMN version INT DEFAULT 0;

-- 查询时拿到当前版本号
SELECT stock, version FROM product WHERE id = 1;
-- 假设查到 stock=100, version=5

-- 更新时带上版本号条件
UPDATE product SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5;
-- 如果 affected rows = 0，说明被别人改过了，重试
```

**另一种乐观锁实现：CAS（Compare And Swap）**

```sql
UPDATE product SET stock = #{newStock}
WHERE id = 1 AND stock = #{oldStock};
```

**对比：**

| | 悲观锁 | 乐观锁 |
|---|---|---|
| 加锁时机 | 读取时就加锁 | 更新时才检查 |
| 并发度 | 低 | 高 |
| 冲突检测 | 数据库层面自动处理 | 应用层面处理 |
| 适用场景 | 写多读少，冲突频繁 | 读多写少，冲突较少 |
| 缺点 | 可能死锁、锁等待 | 冲突多时重试成本高 |

---

### Q3：如何排查和解决死锁？

**答：**

**死锁是什么：** 两个或多个事务互相持有对方需要的锁，形成循环等待，都无法继续执行。

**典型死锁场景：**

```
事务A：锁住行1 → 请求锁住行2
事务B：锁住行2 → 请求锁住行1
```

**MySQL 的死锁检测：**
- InnoDB 默认开启死锁检测（`innodb_deadlock_detect = ON`）。
- 检测到死锁时，InnoDB 会自动回滚持有最少行锁（undo log 最少）的事务，让其他事务继续执行。
- 可以通过 `SHOW ENGINE INNODB STATUS\G` 查看最近一次死锁的详细信息，包括两个事务分别持有什么锁、等待什么锁、执行了什么 SQL。

**排查步骤：**

1. **查看死锁日志：**
   ```sql
   SHOW ENGINE INNODB STATUS\G;
   -- 找到 LATEST DETECTED DEADLOCK 部分
   ```

2. **开启死锁日志持久化：**
   ```sql
   SET GLOBAL innodb_print_all_deadlocks = ON;
   -- 所有死锁信息都会记录到 MySQL 错误日志中
   ```

3. **分析死锁原因：** 关注事务执行的 SQL 语句、锁的类型、涉及的索引。

**解决策略：**

1. **保证加锁顺序一致：** 多个事务以相同的顺序操作表和行，就不会形成循环等待。比如都先操作 id 小的行再操作 id 大的行。

2. **缩短事务：** 事务越短，持锁时间越短，冲突概率越小。把大事务拆成多个小事务。

3. **降低隔离级别：** 从 RR 降到 RC，间隙锁就没有了，能减少很多锁冲突。但需要在应用层处理幻读问题。

4. **为查询加索引：** 没有索引的 UPDATE 会退化为全表扫描加行锁（约等于表锁），极易造成死锁。

5. **设置锁等待超时：** `innodb_lock_wait_timeout`（默认 50 秒），超时自动回滚。

6. **关闭死锁检测（高并发场景）：** 当并发极高时，死锁检测本身也会消耗大量 CPU（O(n²) 复杂度）。可以考虑关闭死锁检测 + 设置较短的锁等待超时时间。`innodb_deadlock_detect = OFF` + `innodb_lock_wait_timeout = 1`。

### Q4：共享锁和排他锁的区别是什么？

**答：**

**共享锁（S Lock，读锁）：**
- 允许事务读取一行数据。
- 多个事务可以同时持有同一行的共享锁（兼容）。
- 通过 `SELECT ... LOCK IN SHARE MODE` 加共享锁。
- 共享锁与共享锁兼容，但共享锁与排他锁互斥。

**排他锁（X Lock，写锁）：**
- 允许事务修改/删除一行数据。
- 同一行数据同时只能有一个排他锁。
- `INSERT`、`UPDATE`、`DELETE` 会自动加排他锁。`SELECT ... FOR UPDATE` 也加排他锁。
- 排他锁与共享锁互斥，排他锁与排他锁也互斥。

**兼容性矩阵：**

| | S（共享锁） | X（排他锁） |
|---|---|---|
| S（共享锁） | 兼容 | 互斥 |
| X（排他锁） | 互斥 | 互斥 |

**意向锁（Intention Lock）：**
- 是表级锁，用来协调行锁和表锁。
- 意向共享锁（IS）：表示事务打算给某些行加共享锁。
- 意向排他锁（IX）：表示事务打算给某些行加排他锁。
- 意向锁由 InnoDB 自动管理，不需要手动操作。它的作用是：当事务想给整张表加表锁时，只需要检查意向锁，不用逐行检查是否有行锁，提高了效率。

### Q5：InnoDB 的锁算法有哪些？

**答：**

InnoDB 在 RR 隔离级别下使用三种锁算法：

1. **Record Lock（记录锁）：**
   - 锁住索引上的一条记录。
   - 只在唯一索引的等值查询时使用。
   - 比如 `WHERE id = 1 FOR UPDATE`，id 是唯一索引，只锁住 id=1 这一行。

2. **Gap Lock（间隙锁）：**
   - 锁住索引记录之间的间隙，不包含记录本身。
   - 比如索引有 id=3 和 id=7，Gap Lock 锁住 (3, 7) 这个开区间。
   - 间隙锁之间不互斥，多个事务可以同时对同一个间隙加 Gap Lock。
   - Gap Lock 只阻止 INSERT 操作。

3. **Next-Key Lock（临键锁）：**
   - Record Lock + Gap Lock，锁住一个左开右闭区间 (a, b]。
   - InnoDB 在 RR 级别下对索引记录的默认加锁方式就是 Next-Key Lock。
   - 比如对 id=7 加 Next-Key Lock，锁住的区间是 (3, 7]。

**退化规则：**
- 唯一索引等值查询：Next-Key Lock 退化为 Record Lock。
- 唯一索引范围查询：仍然使用 Next-Key Lock。
- 普通索引等值查询：使用 Next-Key Lock，但如果右侧记录的索引值不等于查询值，Gap Lock 退化为 Gap Lock（不含右侧记录）。

**加锁案例分析：**

```sql
-- 表 t，id 为主键（唯一索引），有数据 5, 10, 15, 20
BEGIN;
SELECT * FROM t WHERE id = 10 FOR UPDATE;
-- 唯一索引等值查询，只加 Record Lock 在 id=10 上

SELECT * FROM t WHERE id > 10 FOR UPDATE;
-- 范围查询，对 (10, 15], (15, 20], (20, +∞) 加 Next-Key Lock
```

## 五、日志与存储

### Q1：redo log、undo log、binlog 各自的作用是什么？

**答：**

**redo log（重做日志）：**
- **作用：** 保证事务的持久性（Durability）。记录的是"在某个数据页上做了什么修改"（物理日志）。
- **时机：** 事务执行过程中不断写入 redo log buffer，再刷到磁盘（WAL 机制）。
- **特点：** 是 InnoDB 引擎特有的日志；固定大小，循环写入；crash-safe 能力就靠它——即使 MySQL 宕机，重启后也能通过 redo log 恢复已提交的事务。
- **写入方式：** 顺序写磁盘，性能很好。

**undo log（回滚日志）：**
- **作用：** 保证事务的原子性（Atomicity），也是 MVCC 的基础。记录的是数据的逻辑变化（逻辑日志），比如 INSERT 记录对应的 DELETE 操作，UPDATE 记录修改前的旧值。
- **时机：** 事务修改数据之前先把旧数据写入 undo log。
- **特点：** 用于事务回滚（恢复到事务开始前的状态）和 MVCC（通过版本链提供历史版本数据）。undo log 也需要 redo log 来保证持久性（因为 undo log 也是写到数据页中的）。

**binlog（归档日志）：**
- **作用：** 用于数据备份和主从复制。记录所有对数据库执行的逻辑操作（DDL + DML，除了 SELECT）。
- **时机：** 事务提交时写入。
- **特点：** 是 MySQL Server 层的日志，所有存储引擎共用；追加写入，不会覆盖；可以设置过期自动清理；有三种格式（STATEMENT、ROW、MIXED）。
- **主从复制：** 主库写 binlog，从库拉取 binlog 并重放，实现数据同步。

**两阶段提交（保证 redo log 和 binlog 一致性）：**

```
1. Prepare 阶段：将 redo log 写入并标记为 prepare 状态
2. 写 binlog
3. Commit 阶段：将 redo log 标记为 commit 状态
```

如果宕机发生在步骤 1 和 2 之间：redo log 是 prepare 状态但 binlog 不完整，MySQL 重启时会回滚这个事务。如果发生在步骤 2 和 3 之间：binlog 完整，MySQL 会提交这个事务。这样保证了 redo log 和 binlog 的最终一致性。

### Q2：WAL 机制是什么？为什么用 WAL？

**答：**

WAL（Write-Ahead Logging，预写日志）是 InnoDB 的核心写入策略。

**原理：** 当事务提交时，不是直接把修改的数据页刷到磁盘，而是先把修改操作写入 redo log 并刷盘。数据页的修改在内存（Buffer Pool）中完成，后续由后台线程异步刷到磁盘。

**为什么用 WAL：**

1. **顺序写代替随机写：** 数据页在磁盘上的位置是分散的，直接刷数据页是随机 IO，非常慢。redo log 是顺序追加写入，速度远快于随机写。
2. **减少磁盘 IO 次数：** 多个事务对同一个数据页的修改，只需要写一次 redo log，数据页只需要后续批量刷盘一次。
3. **保证 crash-safe：** 即使突然宕机，内存中的数据页丢失了，重启后也可以通过 redo log 重放恢复数据。

**写入流程：**

```
1. 数据修改在 Buffer Pool 中完成（内存操作，很快）
2. 将修改记录写入 redo log buffer（内存）
3. redo log buffer → redo log file（磁盘），通过 fsync 刷盘
4. 后台线程异步将脏页（修改过的数据页）刷到磁盘
```

**刷盘策略（innodb_flush_log_at_trx_commit）：**

| 值 | 行为 | 性能 | 安全性 |
|---|---|---|---|
| 0 | 每秒刷一次 redo log | 最好 | 宕机最多丢 1 秒数据 |
| 1 | 每次提交都 fsync 到磁盘 | 最差 | 不丢数据（默认值，最安全） |
| 2 | 每次提交写到 OS page cache，每秒 fsync | 中间 | 操作系统宕机可能丢数据 |

**面试加分点：** 生产环境通常使用默认值 1，保证数据不丢失。对性能要求高、能容忍少量数据丢失的场景（如日志收集），可以改为 0 或 2。

### Q3：MySQL 主从复制的原理是什么？

**答：**

MySQL 主从复制基于 binlog，分为三个步骤：

**1. 主库（Master）：**
- 主库将数据变更操作写入 binlog。
- 主库启动一个 **Binlog Dump 线程**，负责将 binlog 内容发送给从库。

**2. 从库（Slave）- IO 线程：**
- 从库启动一个 **IO 线程**，连接主库，请求获取 binlog。
- IO 线程接收主库发来的 binlog 内容，写入本地的 **relay log（中继日志）**。

**3. 从库（Slave）- SQL 线程：**
- 从库启动一个 **SQL 线程**，读取 relay log 中的事件，在从库重放这些 SQL 操作，使从库数据与主库保持一致。

**复制类型：**

1. **异步复制（默认）：** 主库写完 binlog 就返回客户端成功，不等从库。性能最好，但主库宕机可能导致从库数据不完整。

2. **半同步复制：** 主库提交事务后，至少等到一个从库确认收到了 binlog 才返回客户端。折中方案，兼顾性能和数据安全。

3. **全同步复制（组复制 Group Replication）：** 所有从库都确认收到并应用了 binlog 才返回。最安全，但性能最差。

**主从延迟的原因和解决方案：**

**原因：** 从库只有一个 SQL 线程重放 relay log，如果主库写入压力大，从库可能跟不上。另外大事务、DDL 操作也会导致延迟。

**解决方案：**
1. 开启**并行复制**（MySQL 5.7+），从库使用多个线程并行应用 relay log（基于库级别或事务级别）。
2. 控制大事务，把大事务拆成小批次。
3. 从库硬件升级（SSD、更多 CPU）。
4. 业务上对一致性要求不高的读请求走从库，要求强一致性的读走主库。

### Q4：binlog 的三种格式有什么区别？

**答：**

| 格式 | 记录内容 | 优点 | 缺点 |
|---|---|---|---|
| **STATEMENT** | 记录原始 SQL 语句 | 日志量小 | 某些函数（NOW()、RAND()）导致主从不一致 |
| **ROW** | 记录每一行数据的变化 | 不会导致主从不一致 | 日志量大（尤其是批量 UPDATE） |
| **MIXED** | 混合模式，普通 SQL 用 STATEMENT，不安全的 SQL 用 ROW | 兼顾两者 | 仍有少数场景不一致 |

**生产环境推荐使用 ROW 格式**，虽然日志量大，但能保证主从数据一致性。配合 binlog 压缩和合理的过期清理策略，可以控制磁盘占用。

**面试加分点：** ROW 格式的 binlog 还可以用于数据恢复——通过 `mysqlbinlog` 工具解析 binlog，可以找回误删的数据。这也是为什么很多公司要求 binlog 必须保留 7 天以上。

### Q5：Buffer Pool 是什么？它是如何工作的？

**答：**

Buffer Pool 是 InnoDB 在内存中维护的数据页缓存池，是 MySQL 性能优化的关键组件。

**作用：** 数据库的读写操作尽量在内存中完成，避免频繁的磁盘 IO。读取数据时，先看 Buffer Pool 中有没有对应的数据页，有就直接返回（缓存命中），没有就从磁盘加载到 Buffer Pool 再返回。写入时，在 Buffer Pool 中修改，形成脏页，后续异步刷盘。

**默认页大小：** 16KB（与磁盘页大小一致）。

**淘汰策略 - 改良的 LRU：**

InnoDB 使用改良的 LRU（Least Recently Used）算法管理 Buffer Pool：

1. 将 LRU 链表分为两部分：**New 区域（约 5/8）** 和 **Old 区域（约 3/8）**。
2. 新加载的数据页先放到 Old 区域的头部，而不是直接放到 New 区域。
3. 在 Old 区域存活超过一定时间（`innodb_old_blocks_time`，默认 1000ms）后再次被访问，才会晋升到 New 区域的头部。
4. 淘汰时从链表尾部（最久未使用）开始淘汰。

**为什么要这么设计：** 防止全表扫描把 Buffer Pool 中的热数据全部刷掉。如果直接用标准 LRU，一次全表扫描就会把所有缓存页替换掉，后续查询全部缓存未命中。

**配置建议：**
- `innodb_buffer_pool_size` 通常设置为物理内存的 60%-75%。
- 生产环境可以设置多个 Buffer Pool 实例（`innodb_buffer_pool_instances`），减少并发访问时的锁竞争。

### Q6：一条 SQL 查询语句在 MySQL 中的完整执行流程是什么？

**答：**

```
客户端发送 SQL
      ↓
  连接器（验证权限、维持连接）
      ↓
  查询缓存（MySQL 8.0 已移除）
      ↓
  解析器（词法分析、语法分析，生成语法树）
      ↓
  预处理器（检查表名、列名是否存在，处理通配符 *）
      ↓
  优化器（选择索引、决定 JOIN 顺序、子查询优化等，生成执行计划）
      ↓
  执行器（调用存储引擎接口，逐行获取数据，返回结果集）
      ↓
  存储引擎（InnoDB）：
    - 检查 Buffer Pool 中是否有目标数据页
    - 没有则从磁盘加载到 Buffer Pool
    - 返回数据行给执行器
      ↓
  结果返回客户端
```

**面试加分点：**

- MySQL 8.0 移除了查询缓存，因为缓存失效太频繁（任何写操作都会导致相关缓存全部失效），命中率低，维护成本反而高。
- 优化器是 MySQL 最复杂的部分，它基于成本模型（Cost-Based Optimizer）选择执行计划，考虑的因素包括：扫描行数、IO 成本、CPU 成本等。
- 执行器在执行前会先检查用户对目标表有没有查询权限。
