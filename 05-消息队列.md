# 消息队列 - 面试知识卡片

> 面向3年经验Java开发者 | 中型互联网公司面试准备

## 一、消息队列基础

### Q1：为什么要使用消息队列？

**答：** 消息队列的核心价值可以总结为三个关键词：**解耦、异步、削峰**。

**解耦：** 假设订单系统下单后需要通知库存系统扣减库存、通知积分系统增加积分、通知物流系统创建运单。如果用同步调用，订单系统和下游系统强耦合，任何一个下游系统故障都会导致下单失败。引入MQ后，订单系统只需将消息发送到MQ，下游系统各自消费，互不影响。

**异步：** 用户注册后需要发送欢迎邮件和短信。如果同步调用邮件和短信服务，假设各耗时200ms，总响应时间就是400ms+。引入MQ后，用户注册逻辑只需20ms写入数据库+5ms发送消息，响应时间大幅降低，邮件和短信异步处理。

**削峰：** 秒杀场景下，瞬时并发可能达到10万QPS，但数据库只能承受1000 QPS。引入MQ后，请求先进入队列，后端服务按照自身能力匀速消费，把瞬时流量高峰拉平成稳定流量。

---

### Q2：消息队列有哪些缺点？

**答：** 引入消息队列会带来以下问题：

**系统可用性降低：** MQ本身可能宕机。如果核心链路依赖MQ，MQ挂了就影响业务。需要部署高可用集群来保障。

**系统复杂度提高：** 引入MQ后需要考虑消息丢失、消息重复、消息顺序性、消息积压等问题，增加了开发和排查问题的复杂度。

**数据一致性问题：** 分布式场景下，生产者发送消息成功但消费者消费失败，会导致数据不一致。需要配合事务消息、补偿机制等来保证最终一致性。

**延迟问题：** 消息从生产到消费有一定延迟，不适合对实时性要求极高的场景。

### Q3：Kafka、RabbitMQ、RocketMQ有什么区别？如何选型？

**答：**

| 特性 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| **开发语言** | Scala/Java | Erlang | Java |
| **吞吐量** | 百万级/秒 | 万级/秒 | 十万级/秒 |
| **延迟** | ms级 | us级 | ms级 |
| **消息模型** | 基于分区的发布订阅 | 基于Exchange的路由 | 基于Topic的发布订阅 |
| **消息可靠性** | 较高 | 高 | 高 |
| **事务消息** | 不支持 | 不支持 | 支持 |
| **延迟消息** | 不原生支持 | 插件支持 | 原生支持 |
| **功能丰富度** | 中等 | 非常丰富 | 丰富 |

**选型建议：**
- **大数据/日志场景：** 选Kafka，吞吐量极高，天然支持分区，适合日志采集、大数据管道。
- **中小规模业务：** 选RabbitMQ，功能丰富，路由灵活，延迟低，社区活跃，Erlang语言对中小团队维护成本略高但使用层面简单。
- **金融/电商核心链路：** 选RocketMQ，支持事务消息、延迟消息，Java开发方便二次开发，阿里系大量验证，适合对消息可靠性要求高的场景。

### Q4：消息队列有哪些常见的消息模型？

**答：** 主要有两种消息模型：

**点对点模型（Queue）：** 一个消息只能被一个消费者消费。多个消费者共同竞争队列中的消息，每条消息只会被其中一个消费者拉取并消费。适合任务分发场景。

**发布/订阅模型（Topic）：** 一个消息可以被多个订阅者消费。生产者将消息发布到Topic，所有订阅该Topic的消费者都能收到消息。适合通知广播、数据同步场景。

RocketMQ和Kafka都同时支持这两种模型。RabbitMQ通过Exchange类型实现更灵活的路由（Direct、Fanout、Topic、Headers四种类型）。

---

### Q5：什么是消息的ACK机制？

**答：** ACK（Acknowledgement）是消费者确认消息已被成功消费的机制。

**自动ACK：** 消费者收到消息后自动确认，不管业务逻辑是否执行成功。优点是吞吐高，缺点是可能丢消息。

**手动ACK：** 消费者在业务逻辑执行成功后，手动向Broker发送确认。如果消费者处理失败或超时未确认，Broker会将消息重新投递给其他消费者。优点是可靠性高，缺点是吞吐略低。

在RocketMQ中，消费者返回`CONSUME_SUCCESS`表示消费成功，返回`RECONSUME_LATER`表示稍后重试。RabbitMQ中通过`basicAck`和`basicNack`来确认和拒绝。

## 二、核心问题

### Q6：如何保证消息不丢失？

**答：** 消息传递涉及三个环节，每个环节都可能丢消息：

**生产者端：**
- 消息发送后Broker没有收到或处理失败。
- 解决方案：开启**发送确认机制**。RocketMQ默认同步发送，返回`SEND_OK`才表示成功。Kafka设置`acks=all`（或`acks=-1`），要求所有ISR副本都写入成功才返回确认。发送失败要**重试**，配合重试策略（指数退避）。

**Broker端：**
- 消息存储在内存中，Broker宕机则消息丢失。
- 解决方案：**持久化到磁盘**。RocketMQ所有消息都写入CommitLog文件，并支持同步刷盘（`flushDiskType=SYNC_FLUSH`）。Kafka消息写入Partition文件，配合多副本机制（`replication.factor >= 3`，`min.insync.replicas >= 2`），保证至少2个副本写入成功。

**消费者端：**
- 消费者自动提交offset后业务处理失败，消息相当于"丢了"（实际是跳过了）。
- 解决方案：**手动提交offset**，业务逻辑执行成功后再提交消费位点。RocketMQ返回`CONSUME_SUCCESS`后才推进offset。Kafka关闭自动提交（`enable.auto.commit=false`），手动调用`commitSync()`。

---

### Q7：如何保证消息的顺序性？

**答：** 消息顺序性分为**全局有序**和**分区有序**。

**Kafka：** 只能保证Partition内有序，不能保证全局有序。要保证同一业务消息有序，需要将相同业务key的消息路由到同一个Partition（通过自定义Partitioner或使用默认的hash分区）。消费端一个Partition只能被消费者组中的一个消费者消费，保证消费顺序。

**RocketMQ：** 发送端通过`MessageQueueSelector`将相同业务ID的消息发到同一个MessageQueue。消费端使用`MessageListenerOrderly`（顺序消费监听器），内部通过加锁保证同一Queue的消息被同一线程顺序消费。

**常见场景：** 订单消息需要保证"创建->支付->发货"的顺序。可以将订单ID作为sharding key，确保同一订单的消息进入同一个队列，消费端顺序处理。

**注意：** 顺序消费会牺牲吞吐量。如果业务不强制要求顺序，尽量不要使用顺序消息。

---

### Q8：如何处理消息积压？

**答：** 消息积压说明生产速度远大于消费速度，需要分紧急处理和长期优化两步走：

**紧急处理：**
1. **快速扩容消费者：** 如果Topic的Partition数足够，增加消费者实例数（不超过Partition数）。
2. **临时扩分区+消费者：** 如果Partition数不够，可以临时创建更多Partition的新Topic，写一个转发程序将原Topic消息分发到新Topic，用更多的消费者并行消费。
3. **跳过非关键消息：** 如果是非核心消息（如日志），可以选择跳过部分消息快速消化积压。

**长期优化：**
1. **优化消费逻辑：** 排查消费慢的原因，是否有慢SQL、外部接口超时、锁竞争等。
2. **批量消费：** 一次拉取多条消息批量处理，减少网络开销。
3. **异步消费：** 消费者内部使用线程池异步处理，提高消费并发度。
4. **增加Partition数：** 提升并行消费的上限。
5. **监控告警：** 设置积压阈值告警，在积压初期就介入处理。

---

### Q9：如何保证消息不被重复消费（幂等性）？

**答：** 消息队列保证的是**at least once**（至少一次投递），所以重复消费是正常现象，关键是在消费端做好**幂等处理**。

**常见幂等方案：**

1. **唯一ID + 数据库去重：** 每条消息携带全局唯一ID（如业务ID或UUID），消费前查询数据库是否已处理。可以用数据库唯一索引保证，重复插入直接报唯一键冲突。

2. **Redis去重：** 消费前用`SET key value NX EX`将消息ID写入Redis，设置过期时间。如果SET失败说明已消费过。适合高并发场景，比数据库查询更快。

3. **状态机校验：** 业务操作前检查当前状态，如订单状态为"已支付"就不再执行支付逻辑。通过前置条件判断来天然保证幂等。

4. **数据库乐观锁：** 使用版本号机制，`UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?`，只有版本号匹配才能更新成功。

5. **Token机制：** 生产者发送消息前先获取Token，消费者消费时校验Token并删除，Token只能使用一次。

### Q10：什么是死信队列？消息过期怎么处理？

**答：**

**死信队列（DLQ, Dead Letter Queue）：** 当消息无法正常消费时，会被转入死信队列。进入死信队列的情况包括：
- 消息被消费者拒绝（Nack/Reject）且不重新入队
- 消息在队列中过期（TTL到期）
- 队列达到最大长度

**RocketMQ的死信队列：** 消费失败的消息会进入重试队列，默认重试16次（间隔递增：10s、30s、1m、2m...2h），全部重试失败后进入死信队列（Topic为`%DLQ%消费者组名`）。死信队列中的消息需要人工排查处理。

**RabbitMQ的死信队列：** 通过给队列配置`x-dead-letter-exchange`参数，死信消息会被路由到指定的Exchange，再进入死信队列。

**消息过期（TTL）：**
- RocketMQ支持在消息属性中设置`delayTimeLevel`来实现延迟投递。
- RabbitMQ可以通过队列级别`x-message-ttl`或消息级别`expiration`设置过期时间。
- Kafka通过Log Retention策略（`log.retention.hours`）控制消息保留时间。

**实际处理：** 对死信队列的消息应该有监控告警，定时检查并人工介入处理。通常需要根据业务决定是重新投递还是丢弃。

---

### Q11：延迟消息如何实现？

**答：** 延迟消息是指消息发送后不立即被消费，而是延迟指定时间后才可被消费。

**RocketMQ原生延迟消息：** 支持固定的延迟级别（18个级别：1s、5s、10s、30s、1m、2m、3m...2h），通过`msg.setDelayTimeLevel(level)`设置。实现原理是Broker收到延迟消息后先存入内部Topic `SCHEDULE_TOPIC_XXXX`，定时任务扫描到期后再转入真正的目标Topic。RocketMQ 5.0版本支持了任意时间延迟。

**RabbitMQ延迟消息：**
- **方案一：TTL + 死信队列。** 消息设置TTL，队列绑定死信Exchange。消息过期后转入死信队列被消费。缺点是有"队头阻塞"问题——如果前面的消息TTL更长，后面的消息即使到期也无法先被消费。
- **方案二：rabbitmq-delayed-message-exchange插件。** 在Exchange层面实现延迟，到期后再路由到队列，更灵活。

**Kafka延迟消息：** Kafka不原生支持延迟消息。常见做法是：使用时间轮算法自建延迟服务，或者使用"时间分区"方案——按延迟时间创建不同的Topic，定时任务扫描到期Topic再转发到目标Topic。

## 三、RocketMQ / Kafka 重点

### Q12：请介绍一下RocketMQ的架构设计

**答：** RocketMQ的架构由四个核心组件构成：

**NameServer：** 轻量级注册中心，提供路由发现服务。每个Broker会向所有NameServer注册自己的信息。NameServer之间互不通信（去中心化设计），简单高效。Producer和Consumer通过NameServer获取Broker的路由信息。

**Broker：** 消息存储和转发的核心服务。分为Master和Slave，Master负责读写，Slave负责备份。消息存储在CommitLog文件中（所有Topic的消息顺序写入同一个文件），通过ConsumeQueue（消费队列）提供按Topic和QueueId的索引查找。

**Producer：** 消息生产者，支持同步发送、异步发送和单向发送。Producer从NameServer获取路由信息后，直接将消息发送到对应的Broker。支持消息分区策略（轮询、哈希等）。

**Consumer：** 消息消费者，支持Push（长轮询拉取，看似推送实为拉取）和Pull（主动拉取）两种模式。消费者从NameServer获取路由信息，然后从Broker拉取消息。

**关键设计亮点：**
- CommitLog顺序写入，利用磁盘顺序IO接近内存随机IO的特性，写入性能极高
- 零拷贝技术（mmap + sendfile）提升文件读取效率
- 消费队列ConsumeQueue是CommitLog的索引，只存储偏移量、大小和Tag哈希值

### Q13：RocketMQ的事务消息是如何实现的？

**答：** RocketMQ的事务消息用于解决分布式事务中"本地事务执行"和"消息发送"的原子性问题。采用**两阶段提交 + 回查机制**：

**流程：**
1. **半消息发送（Half Message）：** Producer发送半消息到Broker，半消息存储在内部Topic `RMQ_SYS_TRANS_HALF_TOPIC`中，此时消费者不可见。
2. **执行本地事务：** Broker返回半消息发送成功后，Producer执行本地事务逻辑（如数据库操作）。
3. **提交或回滚：** 根据本地事务执行结果，Producer向Broker发送Commit或Rollback。Commit则将消息从Half Topic转移到目标Topic，消费者可见；Rollback则删除半消息。
4. **事务回查：** 如果Producer在步骤3异常（如宕机），Broker没有收到Commit或Rollback，则定时（默认60s）回查Producer的事务状态。Producer通过检查本地事务表来确定事务是成功还是失败，返回对应状态。最多回查15次。

**关键点：** 需要配合一张**本地事务表**记录事务状态，回查时通过查询本地事务表来判断事务执行结果。这保证了最终一致性。

### Q14：Kafka为什么吞吐量这么高？

**答：** Kafka的高吞吐来自多个层面的优化：

**1. 顺序写入磁盘：** Kafka将消息顺序追加到Partition文件末尾，利用磁盘顺序IO（约600MB/s）远高于随机IO（约100KB/s）的特性。

**2. 零拷贝技术（Zero Copy）：** 消费消息时使用`sendfile`系统调用，数据从磁盘文件直接通过Socket发送到网络，不经过用户态内存，减少了2次数据拷贝和2次上下文切换。

**3. 分区并行：** 一个Topic分为多个Partition，分布在不同的Broker上，生产和消费可以高度并行。

**4. 批量处理：** Producer可以批量发送消息（`batch.size`配置），减少网络请求次数。Consumer也是批量拉取消息。

**5. 页缓存（Page Cache）：** Kafka不自己管理内存缓存，而是依赖OS的Page Cache。写入消息实际是写入Page Cache，由OS异步刷盘。读取时也大概率命中Page Cache，相当于内存读取。

**6. 数据压缩：** 支持消息压缩（Snappy、LZ4、Zstd），减少网络传输和磁盘IO。可以在Producer端压缩，Broker端存储压缩数据，Consumer端解压。

**综合效果：** 单机吞吐可以轻松达到百万级消息/秒，这也是Kafka在大数据和日志场景被广泛使用的原因。

### Q15：Kafka的分区（Partition）和消费者组（Consumer Group）是什么？

**答：**

**Partition（分区）：** 一个Topic可以分为多个Partition，每个Partition是一个有序的、不可变的消息序列。Partition分布在不同的Broker上，是Kafka实现并行处理和水平扩展的基本单元。每条消息在Partition中有唯一的偏移量（offset）。

Partition的作用：
- 提供并行读写能力
- 实现负载均衡，消息分散到多个Broker
- 是消息顺序性的边界（只保证Partition内有序）

**Consumer Group（消费者组）：** 一组共同消费一个Topic的消费者实例。同一个Consumer Group中，一个Partition只能被组内的一个消费者消费（保证Partition内消息不被重复消费）。

**关键规则：**
- 消费者数 <= Partition数：每个消费者至少消费一个Partition，多余的消费者空闲。
- 消费者数 > Partition数：多出的消费者无法分配到Partition，处于空闲状态。
- 不同的Consumer Group独立消费同一个Topic，互不影响（每个Group各自维护offset），实现广播效果。

**Rebalance（重平衡）：** 当消费者加入或离开Group、Partition数变化时，会触发Rebalance，重新分配Partition给消费者。Rebalance期间消费者无法消费消息（Stop The World），应尽量减少不必要的Rebalance。

### Q16：Kafka如何保证数据不丢失？

**答：**

**Producer端：**
- 设置`acks=all`（或`-1`）：要求所有ISR（In-Sync Replicas）副本都写入成功才返回确认。
- 设置`retries > 0`：发送失败自动重试。
- 设置`max.in.flight.requests.per.connection=1`（Kafka旧版本）或开启幂等Producer（`enable.idempotence=true`），防止重试导致消息乱序。

**Broker端：**
- 设置`replication.factor >= 3`：每个Partition至少3个副本。
- 设置`min.insync.replicas >= 2`：ISR中至少有2个副本。
- 设置`unclean.leader.election.enable=false`：禁止非ISR副本选举为Leader，防止数据丢失。
- 配合`acks=all`和`min.insync.replicas=2`，即使一个Broker宕机也不丢数据。

**Consumer端：**
- 关闭自动提交：`enable.auto.commit=false`。
- 手动提交offset：业务逻辑处理成功后再调用`commitSync()`。
- 合理设置`session.timeout.ms`和`max.poll.interval.ms`，避免消费者被误判下线导致Rebalance和重复消费。

### Q17：RocketMQ如何保证高可用？

**答：**

**NameServer高可用：** 多个NameServer实例组成集群，彼此不通信（去中心化），任意一个宕机不影响服务。Producer和Consumer会连接所有NameServer，一个挂了自动切换。

**Broker高可用：**
- **主从架构（Master-Slave）：** 每个Broker Master配一个Slave，消息从Master同步或异步复制到Slave。Master宕机后，消费者可以从Slave继续消费。
- **DLedger模式：** 基于Raft协议的多副本方案，自动选举Leader，保证数据强一致。副本数通常为3，多数派写入成功才算成功。

**刷盘策略：**
- **同步刷盘（SYNC_FLUSH）：** 消息写入磁盘后才返回成功，数据安全性最高，但吞吐量略低。
- **异步刷盘（ASYNC_FLUSH）：** 消息写入Page Cache后即返回成功，由后台线程异步刷盘。吞吐高但极端情况（宕机+断电）可能丢少量消息。

**生产环境推荐：** DLedger模式 + 同步刷盘，保证数据零丢失和自动故障转移。

### Q18：RocketMQ的消息存储机制是怎样的？

**答：** RocketMQ的存储设计是其高性能的关键：

**CommitLog：** 所有Topic的消息都顺序写入同一个CommitLog文件（默认1GB一个文件）。顺序写入磁盘的性能接近内存随机写入，这是高吞吐的核心。

**ConsumeQueue：** 每个Topic的每个QueueId对应一个ConsumeQueue，它是CommitLog的索引文件。每条记录只存储20字节：CommitLog偏移量（8字节）+ 消息大小（4字节）+ Tag哈希值（8字节）。消费者通过ConsumeQueue快速定位消息在CommitLog中的位置。

**IndexFile：** 消息索引文件，支持通过消息Key或时间范围查询消息。主要用于消息查询和排查问题，不参与正常的消息消费流程。

**消息消费流程：**
1. 消费者从ConsumeQueue读取当前消费位点对应的记录
2. 通过记录中的CommitLog偏移量和消息大小，从CommitLog中读取完整消息
3. 业务处理完成后，更新ConsumeQueue的消费位点

**文件清理：** CommitLog默认保留72小时（可配置），过期文件会被定时任务删除。ConsumeQueue根据CommitLog的清理情况同步清理。

## 附录：面试高频追问

### Q：消息队列选型时你会考虑哪些因素？

**答：** 主要从以下维度评估：
1. **业务场景：** 日志/大数据选Kafka，核心业务链路选RocketMQ，中小系统选RabbitMQ。
2. **吞吐量需求：** 百万级选Kafka，十万级选RocketMQ，万级选RabbitMQ。
3. **消息可靠性：** 是否需要事务消息（RocketMQ），是否允许少量丢失。
4. **功能需求：** 延迟消息、死信队列、消息过滤、消息回溯等。
5. **团队技术栈：** Java团队优先RocketMQ，方便二次开发和问题排查。
6. **社区和生态：** 社区活跃度、文档完善度、公司是否有成熟的运维经验。

### Q：消费者处理消息很慢怎么办？

**答：** 排查思路：
1. 先确认是消费慢还是生产快导致的积压。
2. 检查消费端是否有慢SQL、外部接口调用超时、频繁GC等。
3. 优化消费逻辑后，考虑增加消费者实例数或Topic Partition数来水平扩展。
4. 消费端内部使用线程池异步处理，将消息接收和业务处理解耦。
5. 如果业务允许，可以采用批量消费减少网络请求开销。
